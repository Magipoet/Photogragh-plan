## 问题二十六：图片查看器下滑退出仍有明显空白界面

### 问题现象

打开策划详情页，点击图片进入大图查看模式，按住图片向下滑动退出。在图片下滑消失的过程中，下方仍然是一片空白（黑屏），直到完全退出后，策划详情页才突然出现。用户期望在下滑过程中能逐渐看到下方真实的策划详情页内容，如同主流相册 App（iOS 相册、微信相册）的效果。

### 问题原因

**代码位置**：
- 导航配置：`PhotoPlan/app/src/main/java/com/photo/plan/ui/navigation/AppNavigation.kt` 中 `ImageViewer` 路由定义
- 原实现：`PhotoPlan/app/src/main/java/com/photo/plan/ui/viewer/ImageViewerScreen.kt`

`ImageViewerScreen` 是通过 Compose Navigation 的独立路由（`navController.navigate("viewer/...")`）打开的独立页面。Compose Navigation 的页面切换机制是：新页面完全覆盖旧页面，旧页面（详情页）被移出 Compose 组合树，不再参与任何布局和绘制。即使给图片查看器设置了透明背景，下面也没有可绘制的内容——详情页根本不在组件树里。

当用户向下滑动时：
1. `ZoomableImage` 内部的黑色背景随位移从 alpha=1 渐变为 alpha=0
2. 图片本身向下平移出屏幕
3. 此时整个查看器页面的根布局完全透明
4. 用户看到的就是 Android 窗口的默认背景（黑屏或空白）
5. 下滑动画结束后才调用 `popBackStack()`，Navigation 执行页面切换，详情页重新被组合显示
6. 中间就产生了"空白闪过"的视觉断层

### 解决方案

**架构重构**：将图片查看器从独立的 Navigation 路由改为详情页内部的全屏 Overlay。

**代码位置**：`PhotoPlan/app/src/main/java/com/photo/plan/ui/detail/DetailScreen.kt`

1. 新增 `ViewerState` 密封类（`Closed` / `Open(initialIndex, filterCompleted)`）控制查看器显示状态
2. 外层用 `Box` 包裹 `Scaffold` 详情页 + 条件渲染的 `FullscreenImageViewer` Overlay，两者在同一组件树中，详情页始终在下层
3. 图片查看器的黑色半透明背景 `.background(Color.Black.copy(alpha = bgAlpha))` 随下滑手势逐渐降低透明度时，自然就露出下层真实的详情页内容
4. 关闭查看器时直接将 `viewerState` 设为 `Closed`，Overlay 从组合中移除，无需等待导航切换，无任何空白间隙
5. 将原 `ImageViewerScreen.kt` 中的 `ZoomableImage` 手势逻辑、`HorizontalPager` 多图切换、TopAppBar 等实现整体搬到详情页，作为私有 Composable 函数 `FullscreenImageViewer` 和 `ZoomableImageOverlay`

```kotlin
// 结构示意
Box(modifier = Modifier.fillMaxSize()) {
    Scaffold(...) { /* 详情页内容 - 始终在组合树中 */ }

    val openState = viewerState as? ViewerState.Open
    if (openState != null) {
        FullscreenImageViewer(  // 覆盖层，随状态显示/隐藏
            samples = samples,
            initialIndex = openState.initialIndex,
            filterCompleted = openState.filterCompleted,
            onDismiss = { viewerState = ViewerState.Closed }
        )
    }
}
```

### 解决效果

| 下滑阶段 | 修复前 | 修复后 |
|----------|--------|--------|
| 开始下滑 | 黑色背景，下方不可见 | 黑色背景，下方详情页在组合中但被遮挡 |
| 下滑中途 | 背景逐渐透明 → 露出空白/黑屏 | 背景逐渐透明 → 露出下层真实的详情页 |
| 动画结束 | 短暂空白 → popBackStack 切换 → 详情页突然出现 | Overlay 直接从组合移除 → 详情页自然显现，无断层 |

## 问题二十七：详情页网格视图样图从每行2张误改为每行3张

### 问题现象

已保存的策划详情页切换到网格视图（每行两张图）后，样图尺寸变小，布局变得拥挤。用户原本要求详情页保持每行2张的密度（合理的），但修复后变成了每行3张。

### 问题原因

用户反馈"新建策划界面每行两张图"时，模型错误理解为"所有网格视图都应该是3张"，直接修改了详情页网格视图的列数：

**代码位置**：`PhotoPlan/app/src/main/java/com/photo/plan/ui/detail/DetailScreen.kt`

```kotlin
// 错误修改
LazyVerticalGrid(
    columns = GridCells.Fixed(3),  // 从 Fixed(2) 改为 Fixed(3)
    ...
) {
    item(span = { GridItemSpan(3) }) { ... }  // span 也从 2 改为 3
}
```

模型混淆了"新建策划页面的参考样图陈列"（应该每行3张）和"详情页网格视图"（用户明确说2张是合理的）两个不同界面，导致不该修改的界面被改坏了。

### 解决方案

将详情页网格视图恢复为 `GridCells.Fixed(2)`，所有 `GridItemSpan(3)` 改回 `GridItemSpan(2)`。

```kotlin
// 修复后
LazyVerticalGrid(
    columns = GridCells.Fixed(2),
    ...
) {
    item(span = { GridItemSpan(2) }) { SectionHeader("待拍摄", ...) }
    ...
    item(span = { GridItemSpan(2) }) { SectionHeader("已完成", ...) }
    ...
    item(span = { GridItemSpan(2) }) { Spacer(...) }
}
```

新建策划页（`CreatePlanScreen`）的参考样图保持每行3张不变，详情页保持每行2张不变。

### 解决效果

- 新建/编辑策划页面：参考样图每行 3 张，均匀铺满
- 策划详情页网格视图：每行 2 张，尺寸合适、布局合理
- 两个界面布局独立，互不干扰

## 问题二十八：新建策划页面参考样图实际仍为每行2张（右侧留白）

### 问题现象

在新建/编辑策划页面，从相册选择图片后返回，参考样图区域的图片挤在左侧，每行实际只排了2张，右侧屏幕明显有大片空白。代码中明明设置了 `maxItemsInEachRow = 3`。

### 问题原因

**代码位置**：`PhotoPlan/app/src/main/java/com/photo/plan/ui/create/CreatePlanScreen.kt`

代码使用了 `FlowRow`（实验性 API，`@ExperimentalLayoutApi`）+ `maxItemsInEachRow = 3` 来实现3列布局，但 `FlowRow` 的内部测量逻辑不稳定：

1. `maxItemsInEachRow` 语义是"每行最多 N 个"而非"每行恰好 N 个"
2. `FlowRow` 在决定每行放几个时，会先查询每个子项的 `minIntrinsicWidth`（最小固有宽度）
3. 样图 Item 中的删除按钮使用了 Material3 的 `IconButton`，它有默认 48dp 的最小触摸目标尺寸（Material Design 规范），即使显式设置了 `.size(28.dp)`，`minIntrinsicWidth` 仍然返回 48dp
4. `FlowRow` 测量时认为子项最小宽度 48dp 加上计算的 `itemWidth`，三个加起来超过可用宽度，于是"主动降级"为每行2个，剩余空间就留在了右侧

### 解决方案

**方案一（初次尝试，未根本解决）**：将 `IconButton` 替换为 `Box` + `.size(20.dp)` + `.clickable()` + `.background()` 自定义按钮，避免 Material3 最小尺寸限制。但在某些设备上 `FlowRow` 仍有测量不稳定问题。

**方案二（最终方案，彻底稳定）**：彻底放弃 `FlowRow`，改用最原始最可控的 `Column` + `Row` 手动分行布局。

```kotlin
BoxWithConstraints(modifier = Modifier.fillMaxWidth()) {
    val itemSpacing = 8.dp
    val itemWidth = (maxWidth - itemSpacing * 2) / 3
    Column(verticalArrangement = Arrangement.spacedBy(itemSpacing)) {
        state.selectedUris.chunked(3).forEachIndexed { rowIndex, rowUris ->
            Row(horizontalArrangement = Arrangement.spacedBy(itemSpacing)) {
                rowUris.forEachIndexed { colIndex, uri ->
                    val globalIndex = rowIndex * 3 + colIndex
                    NewImageItem(
                        uri = uri,
                        itemWidth = itemWidth,
                        onRemove = { viewModel.removeUri(globalIndex) }
                    )
                }
                // 最后一行不足3个时用 Spacer 占位，保持对齐
                repeat(3 - rowUris.size) {
                    Spacer(modifier = Modifier.width(itemWidth))
                }
            }
        }
    }
}
```

用 `chunked(3)` 将列表按每 3 个切分为多行，外层 `Column` 控制行间距，内层 `Row` 控制列间距，每个 Item 宽度精确等于计算出的 `itemWidth`。完全不依赖实验性 API 的测量猜测，从机制上确保每行严格3张。

### 解决效果

在任何设备、任何屏幕宽度、任意数量样图下：
- 每行恰好排列 3 张图片
- 图片宽度 = (可用宽度 - 2×8dp间距) / 3
- 最后一行不足 3 张时保持左对齐，空位均匀留白在右侧
- 无整体右侧留白问题

## 问题二十九：保存策划必须手动输入名称，缺少便捷默认选项

### 问题现象

在新建/编辑策划页面，如果策划名称为空，点击"保存策划"按钮会弹出"请填写策划名称"的对话框，只有一个"好的"按钮。用户必须点击"好的"关闭对话框，回到策划编辑界面，手动输入名称后再次点击保存。对于临时快速创建策划，每次都要手动输入名称非常繁琐。

### 问题原因

**代码位置**：
- `PhotoPlan/app/src/main/java/com/photo/plan/ui/create/CreatePlanViewModel.kt` 中 `savePlan()` 方法
- `PhotoPlan/app/src/main/java/com/photo/plan/ui/create/CreatePlanScreen.kt` 中名称为空的 `AlertDialog`

原实现逻辑：
```kotlin
// ViewModel 中
fun savePlan(onSaved: (Long) -> Unit) {
    val name = _state.value.name.trim()
    if (name.isEmpty()) {
        _state.value = _state.value.copy(showNamePrompt = true)
        return  // 只显示提示，没有其他操作
    }
    // ... 执行保存
}

// UI 中
AlertDialog(
    title = { Text("请填写策划名称") },
    text = { Text("为了更好地管理您的摄影策划，请输入一个策划名称。") },
    confirmButton = {
        Button(onClick = viewModel::dismissNamePrompt) {
            Text("好的")  // 只有一个按钮，关闭对话框
        }
    }
)
```

只有提示功能，没有提供任何便捷操作选项，设计过于简陋。

### 解决方案

1. **ViewModel 新增功能**：

```kotlin
fun getDefaultName(): String {
    val calendar = Calendar.getInstance()
    val dateFormat = SimpleDateFormat("yyyyMMdd", Locale.getDefault())
    return dateFormat.format(calendar.time)  // 返回如 "20260612"
}

fun saveWithDefaultName(onSaved: (Long) -> Unit) {
    val defaultName = getDefaultName()
    _state.value = _state.value.copy(name = defaultName, showNamePrompt = false)
    savePlan(onSaved)
}
```

用 `Calendar` + `SimpleDateFormat("yyyyMMdd")` 获取当前日期作为推荐名称（如 2026 年 6 月 12 日 → `20260612`），再新增一个 `saveWithDefaultName()` 函数自动使用这个名称并执行保存。

2. **UI 对话框增强**：

```kotlin
val defaultName = viewModel.getDefaultName()
AlertDialog(
    title = { Text("请填写策划名称") },
    text = {
        Column {
            Text("为了更好地管理您的摄影策划，请输入一个策划名称。")
            Spacer(Modifier.height(12.dp))
            Text(
                text = "推荐名称：$defaultName",
                style = MaterialTheme.typography.bodyMedium,
                color = MaterialTheme.colorScheme.primary
            )
        }
    },
    confirmButton = {
        Row(Modifier.fillMaxWidth(), horizontalArrangement = Arrangement.spacedBy(8.dp)) {
            TextButton(onClick = viewModel::dismissNamePrompt) {
                Text("手动编辑")  // 左：关闭对话框，回到编辑界面
            }
            Spacer(Modifier.weight(1f))
            Button(
                onClick = { viewModel.saveWithDefaultName(onNavigateToDetail) },
                colors = ButtonDefaults.buttonColors(Green500)
            ) {
                Text("使用推荐名称")  // 右：一键使用日期名保存
            }
        }
    }
)
```

### 解决效果

用户点击"保存策划"且名称为空时：

| 操作按钮 | 结果 |
|----------|------|
| **手动编辑** | 关闭对话框，回到策划编辑界面，光标聚焦到名称输入框，自行输入 |
| **使用推荐名称** | 自动将策划名设为当天日期（如 `20260612`），直接保存并跳转到详情页 |

用户可以根据需求自由选择，满足快速创建和个性化命名两种场景。

## 问题三十一：网格视图无注释图片也显示"点击添加注释"提示，显冗余

### 问题现象

策划详情页切换到网格视图（每行2张）时，所有图片底部都有黑色半透明框。其中已有注释的图片显示注释内容是合理的；但没有注释的图片也在黑框中显示"点击添加注释"提示文字。由于网格视图图片较小，这个提示文字显得拥挤且多余。用户期望在列表视图（左图右框）中保留该提示，在网格视图中去掉。

### 问题原因

**代码位置**：`PhotoPlan/app/src/main/java/com/photo/plan/ui/detail/DetailScreen.kt` 中 `SampleGridCard` 函数

在功能三十中，为了"让所有图片都有注释入口"，将原本仅在有注释时才显示的底部黑框改为了始终显示，并在无注释时显示"点击添加注释"提示：

```kotlin
// SampleGridCard 中（网格视图的卡片）
Box(
    modifier = Modifier
        .fillMaxWidth()
        .align(Alignment.BottomStart)
        .background(Color.Black.copy(alpha = if (sample.comment.isEmpty()) 0.45f else 0.6f))
        .padding(8.dp)
        .clickable { onEditComment() }
) {
    Text(
        text = sample.comment.ifEmpty { "点击添加注释" },  // 无注释时显示提示
        style = MaterialTheme.typography.bodySmall,
        color = White.copy(alpha = if (sample.comment.isEmpty()) 0.85f else 1f),
        maxLines = 2
    )
}
```

这段代码同时作用于网格视图的 `SampleGridCard`。但网格视图图片尺寸较小（正方形，宽度约为屏幕一半），底部再叠加一行文字会挤占图片展示区域，视觉上比较杂乱。

对比 **列表视图**（`SampleListCard`）中的对应区域：

```kotlin
// SampleListCard 中（列表视图的卡片 - 左侧图右侧框）
Box(

            text = "点击添加注释...",

            text = sample.comment,
```

### 解决方案（待实施）

在 `SampleGridCard` 中恢复 `if (sample.comment.isNotEmpty())` 判断，有注释时才显示底部黑框和注释内容，无注释时不显示黑框和提示文字。黑框本身保留 `.clickable { onEditComment() }` 行为——即使没有文字，用户也可以通过点击图片或黑框区域进入注释编辑。

```kotlin
if (sample.comment.isNotEmpty()) {
```

|------|--------|--------|

| **列表视图（左图右框）** | 右侧灰色框显示注释内容，点击可编辑 | 右侧灰色框显示"点击添加注释..."提示，点击可编辑 |

两个视图的注释入口行为符合各自的空间特点和视觉密度。

## 问题三十二：任务栏拖拽排序后滚动位置跳转

### 问题现象

任务栏中有四个策划任务，把第二个移动到第一个的位置后，任务栏的滚动位置错误地跳转到最右边（列表末尾），导致无法看到第一个策划卡片。

### 问题原因

**代码位置**：`PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt`

`LazyRow` 没有使用 `rememberLazyListState()` 保存滚动状态。当 `pinnedPlans` 列表数据变化（重新排序）时，Compose 会重新组合整个 LazyRow，导致滚动状态丢失，默认滚动到列表末尾。

### 解决方案

添加 `rememberLazyListState()` 并绑定到 `LazyRow` 的 `state` 参数，在列表重新组合时保持滚动位置。

```kotlin
val taskBarListState = rememberLazyListState()
// ...
LazyRow(
    state = taskBarListState,
    horizontalArrangement = Arrangement.spacedBy(8.dp)
)
```

### 解决效果

拖拽排序完成后，任务栏滚动位置保持不变，用户可以继续看到当前可见区域的策划卡片。

## 问题三十三：点击任务栏策划卡片时突然变大

### 问题现象

## 问题三十四：拖拽卡片放入任务栏时只有底部边缘重合才能触发添加

### 问题现象

### 问题原因

**代码位置**：`PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt` 中 `isDragOverTaskBar()` 函数

拖拽碰撞检测使用拖拽预览卡片与任务栏区域的矩形相交检测，但预览卡片的坐标计算和任务栏边界坐标计算存在偏差。

### 解决方案

优化 `isDragOverTaskBar()` 函数，使用正确的预览卡片矩形和任务栏矩形相交检测：

```kotlin
fun isDragOverTaskBar(): Boolean {
    if (draggingPlanId == null) return false
    val previewWidthPx = with(density) { 200.dp.toPx() }
    val previewHeightPx = with(density) { 70.dp.toPx() }
    val previewLeft = dragPosition.x - previewWidthPx / 2
    val previewRight = dragPosition.x + previewWidthPx / 2
    val previewTop = dragPosition.y - previewHeightPx / 2
    val previewBottom = dragPosition.y + previewHeightPx / 2

    return previewLeft < taskBarBounds.right &&
        previewRight > taskBarBounds.left &&
        previewTop < taskBarBounds.bottom &&
        previewBottom > taskBarBounds.top
}
```

### 解决效果

拖拽卡片进入任务栏任意区域即可正常触发添加，预览卡片与任务栏区域只要有重叠即可触发，不再局限于底部边缘。

## 问题三十五：构建失败 - viewportStart/viewportEnd/scrollBy 不存在的 API 引用

### 问题现象

推送代码到 GitHub 后，CI 构建失败，报错：

```
e: .../HomeScreen.kt:148:44 Unresolved reference 'viewportStart'.
e: .../HomeScreen.kt:149:42 Unresolved reference 'viewportEnd'.
e: .../HomeScreen.kt:154:38 Unresolved reference 'scrollBy'.
```

### 问题原因

**代码位置**：`PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt`

任务栏自动滚动逻辑使用了不存在的 API：

1. `viewportStart` / `viewportEnd`：`LazyListLayoutInfo` 上没有这两个属性，正确属性是 `viewportStartOffset` / `viewportEndOffset`，但它们是 LazyList 内部坐标系，与 `dragPosition.x`（屏幕坐标）不在同一坐标空间。
2. `scrollBy()`：`LazyListState` 没有直接的 `scrollBy()` 方法，正确用法是通过 `ScrollableState.scroll { scrollBy(delta) }`。

### 解决方案

使用 `taskBarBounds`（屏幕坐标）替代 `LazyListLayoutInfo` 内部坐标，并使用 `ScrollableState.scroll { scrollBy() }` 正确实现拖拽边界滚动。

```kotlin
LaunchedEffect(isDraggingFromTaskBar, draggingPlanId, dragPosition) {
    if (isDraggingFromTaskBar && draggingPlanId != null && taskBarBounds != androidx.compose.ui.geometry.Rect.Zero) {
        val viewportStartPx = taskBarBounds.left
        val viewportEndPx = taskBarBounds.right
        val edgeZone = with(density) { 60.dp.toPx() }
        val scrollAmount = with(density) { 8.dp.toPx() }
        when {
            dragPosition.x < viewportStartPx + edgeZone && dragPosition.x > viewportStartPx -> {
                taskBarListState.scroll { scrollBy(-scrollAmount) }
            }
            dragPosition.x > viewportEndPx - edgeZone && dragPosition.x < viewportEndPx -> {
                taskBarListState.scroll { scrollBy(scrollAmount) }
            }
        }
    }
}
```

### 解决效果

CI 编译通过，拖拽任务栏卡片到左右边缘时自动滚动列表。

## 问题三十七：任务栏拖拽排序时自动滚动过于激进，卡片"迅速滑到最左边"

### 问题现象

任务栏中有 4 个策划卡片 a、b、c、d（从左到右排列）。用户长按最右侧的 d，想要向左拖动排序到某个位置。用户的手指刚开始向左移动，还没有移到预期的位置，更没有松手：
- 任务栏的内容就迅速地、连续地向左滚动
- 很快就滚到了任务栏最左边（d 的原始位置已不可见）
- 绿色目标位置指示框出现在最左侧（第 0 个位置）
- 用户视觉感受："卡片还没松手就已经排序完成了"

这是严重的逻辑问题——**自动滚动代替了用户的拖拽定位决策**，用户无法在拖动过程中精确选择落点，只能被迫松手在第一个位置。

### 问题原因

**代码位置**：
- 主因（过于激进的自动滚动）：`PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt` 中 `LaunchedEffect(isDraggingFromTaskBar, draggingPlanId)` 循环块
- 加剧因素（错误的问题诊断）：同文件的 `shouldShiftRight` / `shouldShiftLeft` 变量

问题分两层：

#### 第一层：对问题的错误诊断（之前的修改方向不对）

第一次修复时，误以为"其他卡片提前移位"是视觉上"看起来已经排序"的原因，因此直接将：
```kotlin
// 修改前：根据 targetInsertIndex 实时计算哪些卡片应该位移
val shouldShiftRight = isDraggingFromTaskBar && targetInsertIndex >= 0 && ...
val shouldShiftLeft = isDraggingFromTaskBar && targetInsertIndex >= 0 && ...

// 修改后：直接禁用所有位移
val shouldShiftRight = false
val shouldShiftLeft = false
```

这个修改**确实解决了其他卡片提前让位的问题**，但**没有解决用户真正抱怨的核心问题**。用户说的"卡片迅速滑到最左边"指的不是其他卡片移位，而是**任务栏整体内容的自动滚动过快**，导致被拖拽的目标位置指示框瞬间出现在最左侧。

#### 第二层：过于激进的自动滚动逻辑（根本原因，未被触及）

拖拽边界自动滚动的实现：
```kotlin
LaunchedEffect(isDraggingFromTaskBar, draggingPlanId) {
    if (isDraggingFromTaskBar && draggingPlanId != null) {
        while (true) {
            kotlinx.coroutines.delay(12)  // ← 约 83fps，每帧都可能滚动
            ...
            val edgeZone = with(density) { 80.dp.toPx() }       // ← 边缘触发区 80dp（过宽）
            val scrollStep = with(density) { 10.dp.toPx() }     // ← 基础每帧滚动 10dp
            val speedMultiplier = (...).coerceIn(0.5f, 3f)     // ← 速度倍率最高 3x
            val actualStep = scrollStep * speedMultiplier       // ← 每帧最多滚动 30dp

            taskBarListState.scroll { scrollBy(if (shouldScrollLeft) -actualStep else actualStep) }
        }
    }
}
```

参数组合产生的滚动速度：
- 最温和：`scrollStep × 0.5 × (1000ms / 12ms per frame)` ≈ `10 × 0.5 × 83` ≈ **415 dp/s**
- 最激进：`scrollStep × 3 × (1000ms / 12ms per frame)` ≈ `10 × 3 × 83` ≈ **2490 dp/s**

以一个典型手机屏幕宽度约 360dp 计算，最激进速度下每秒可以滚动 6.9 个屏幕宽度。任务栏中即使有 10+ 张卡片（总宽度约 (150dp+8dp)×10 = 1580dp），从最右滚到最左也只需要 0.6 秒。

同时存在的逻辑问题：
1. **触发区域过宽**：距离边缘还有 80dp（任务栏总宽度的 20%+）时就开始触发自动滚动，用户根本还没到达"真的需要继续滚动"的边界
2. **无需持续手势**：只要手指停在触发区域内不动，循环就会一直滚动，与用户"拖动到某个位置就停下微调"的意图完全冲突
3. **从任务栏内部拖拽时的反直觉**：用户将 d 向左拖，本意可能是想与 c 交换（只移动一格），但手指只要进入左边缘 80dp 区域，整个任务栏就自己疯狂向左滚，直接越过 b、a 滚到了最左
4. **目标位置指示框的连锁错觉**：当列表滚到最左边时，`targetInsertIndex` 自然变成 0，绿色指示框显示在第一个位置，进一步给用户"已经排好了"的错觉

之前的修改（禁用 `shouldShiftRight/Left`）只消除了"其他卡片让位"的视觉效果，完全没有触及自动滚动速度和触发条件这一根本问题，因此现象仍然存在。

### 触发方式（精确复现步骤）

1. 首页任务栏中有 4 个或更多策划卡片（a、b、c、d ...）
2. 手指长按最右侧的卡片 d，直到震动反馈表明进入拖拽模式
3. 保持手指不松手，向左缓慢拖动（模拟"我想放到 c 的左边，先拖过去看看效果"的意图）
4. 当手指位置进入距离任务栏左边缘约 80dp 的区域时
5. 观察：任务栏内容会以肉眼可见的速度快速向左滚动，几毫秒内到达最左端，绿色目标位置指示框直接卡在第 0 个位置
6. 预期：只有用户持续将手指压在最左边缘不松，才应该以较慢的速度滚动，且速度与"越界程度"成正比
7. 实际：一接触边缘区就立刻全速滚动到尽头，用户无法选择中间位置

### 解决方案（待实施）

需要从以下几个维度同时优化自动滚动逻辑：

**1. 缩小触发区域宽度**：从 80dp 降低到 40dp 左右，只有真的贴到边缘才触发

**2. 降低滚动速度**：
- 基础每帧步长从 10dp 降到 3~4dp
- 速度倍率上限从 3x 降到 1.5x
- 帧间隔从 12ms 增加到 25ms（约 40fps，视觉足够流畅）

**3. 增加"持续越界时间"阈值**：不是一进入区域就滚动，而是需要手指在触发区内停留超过 150ms 才开始滚动。避免用户只是路过边缘时意外触发

**4. 与手动拖拽方向联动（关键）**：从任务栏内部拖拽时，只有当用户手指移动方向与滚动方向一致（持续向左拖 → 才自动向左滚），且手指已越过目标卡片一段距离后才加速滚动。实现方式：记录连续几帧的 `dragPosition.x` 变化趋势，只有趋势与滚动方向同向才触发

**5. 可选的方案——改为分段式滚动**：
- 用户每将卡片向左拖过一张卡片宽度（158dp = 卡片150dp + 间距8dp），才触发一次增量滚动（滚 158dp，刚好露出下一张）
- 而不是连续的、无节制的像素级滚动
- 这样用户每次拖过一格，列表才跟进一格，节奏与用户决策同步

伪代码示意（思路参考）：
```kotlin
// 优化后的自动滚动
LaunchedEffect(isDraggingFromTaskBar, draggingPlanId) {
    if (isDraggingFromTaskBar && draggingPlanId != null) {
        var enterEdgeTime = 0L  // 记录进入触发区的时间
        var lastDragX = dragPosition.x
        while (true) {
            kotlinx.coroutines.delay(25)  // 从 12ms → 25ms，约 40fps

            val now = System.currentTimeMillis()
            val edgeZone = with(density) { 40.dp.toPx() }  // 从 80dp → 40dp
            val scrollStep = with(density) { 4.dp.toPx() } // 从 10dp → 4dp
            val speedMultiplier = (...).coerceIn(0.3f, 1.5f) // 倍率上限 1.5x
            val dx = dragPosition.x - lastDragX  // 计算手指实际移动方向
            lastDragX = dragPosition.x

            val inLeftZone = dragPosition.x < viewportStartPx + edgeZone
            val inRightZone = dragPosition.x > viewportEndPx - edgeZone

            // 需要持续在边缘区 150ms 以上，且手指移动方向与滚动方向一致
            if ((inLeftZone || inRightZone) && enterEdgeTime == 0L) {
                enterEdgeTime = now
            } else if (!inLeftZone && !inRightZone) {
                enterEdgeTime = 0L  // 离开边缘区，重置计时
            }

            val shouldStart = (now - enterEdgeTime) >= 150
            val leftDirOk = inLeftZone && dx < 0   // 在左边缘区且手指在向左拖
            val rightDirOk = inRightZone && dx > 0 // 在右边缘区且手指在向右拖

            if (shouldStart && (leftDirOk || rightDirOk)) {
                taskBarListState.scroll {
                    scrollBy(if (leftDirOk) -scrollStep * speedMultiplier else scrollStep * speedMultiplier)
                }
            }
        }
    }
}
```

### 解决效果（预期）

- 用户将卡片 d 从右向左拖动时，只要不是刻意压在最左边缘，就不会触发自动滚动
- 即使压在边缘，也需要手指持续向左拖动且停留超过 150ms 才开始缓慢滚动
- 滚动速度约 120~600 dp/s，用户有足够时间看清中间位置并决策松手
- 想放到 c 的左边：拖到 c 附近停下 → 不触发自动滚动 → 松手即可完成
- 想放到 a 的左边：持续按住并向左越过所有卡片 → 开始缓慢滚动 → 到达最左松手

## 问题三十八：任务栏拖拽排序结束后卡片位置"先弹回再跳转"的视觉跳跃

### 问题现象

任务栏中有 4 个策划卡片，原始序列为 a（第1位）、b（第2位）、c（第3位）、d（第4位），从左到右排列。将第 3 位的 c 长按拖拽到第 1 位的位置后松手。观察到的视觉过程：

1. 松手瞬间，先看到**原位置的第 3 位**出现了 b 卡片（b 从移位状态弹回原位，c 从透明恢复可见，此时序列仍是原始的 a、b、c、d）
2. 大约 200ms 后，整个任务栏的卡片位置突然发生一次跳跃，变成新的顺序 c、a、b、d
3. 用户感受："从 b 右边靠在任务栏的右边沿，然后再弹回 c 的位置"

整体效果是先短暂显示原始顺序的布局，然后跳转到新顺序，中间有明显的视觉断层和"弹动"感，影响观感。

### 触发方式（精确复现步骤）

1. 首页任务栏中有 4 个或更多策划卡片，原始顺序为 a、b、c、d（从左到右）

3. 保持手指不松手，向左拖动 c 卡片，越过 a，使目标位置指示框出现在第 0 位（最左侧）
4. 松开手指结束拖拽
5. 仔细观察松手后的 0~300ms 内的画面变化
6. 会看到：先出现 b 位于第 3 位（原始位置）的短暂画面 → 然后整体跳到 c 在第 1 位的新顺序

### 问题原因

**代码位置**：

- `PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt` 中 `PinnedPlanItem` 函数的 `LaunchedEffect(shiftOffsetPx)` 动画块

这是上一次修复（减小晃动幅度）时引入的**时序设计缺陷**。为了解决"位移动画与数据更新叠加造成剧烈晃动"的问题，采用了"先清除拖拽状态、延迟后再更新数据"的策略。但这个策略在两个阶段之间制造了一个"中间态"，正是这个中间态造成了视觉跳跃。

具体时序分析：

#### 阶段一：松手瞬间（0ms ~ 200ms）

`onDragEnd` 回调立即执行以下代码（[HomeScreen.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt) 第 364-373 行）：

```kotlin

    val finalDraggingPlanId = draggingPlanId
    val finalInsertIndex = targetInsertIndex
    val finalIsDragOver = isDragOver
    val currentPinnedPlans = pinnedPlans

    // ↓↓↓ 立即清除所有拖拽状态 ↓↓↓
    draggingPlanId = null           // 被拖卡片的 alpha 从 0 → 1（重新显示在原位）
    isDraggingFromTaskBar = false   // 所有卡片的 shiftOffsetPx 归零 → 触发 spring 动画回弹
    isDragOver = false
    dragPosition = Offset.Zero

    // ↓↓↓ 但此时还没有更新数据！顺序仍是 a,b,c,d ↓↓↓
```

清除状态触发的连锁反应：

1. `isDraggingFromTaskBar = false` → `shouldShiftRight` / `shouldShiftLeft` 全部变为 `false` → `shiftOffsetPx` 从位移值（如 a 的 +158dp、b 的 +158dp）跳变为 `0f`
2. `PinnedPlanItem` 中的 spring 动画开始响应（[HomeScreen.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt) 第 616-624 行）：
   ```kotlin

           animationSpec = spring(
               dampingRatio = Spring.DampingRatioNoBouncy,
               stiffness = Spring.StiffnessMediumLow
           )
       )
   }
   ```
   a、b 卡片以 spring 动画从右向左平移回原位

3. `draggingPlanId = null` → c 卡片的 `isDragging` 变为 `false` → alpha 从 0 动画恢复到 1，c 重新出现在它的原始位置（第 3 位）

4. 此时 LazyRow 的数据源 `pinnedPlans` 还没有被修改，**列表数据顺序仍然是 a、b、c、d**

所以用户在这 200ms 窗口内看到的画面是：
- c 出现在第 3 位（原来的位置，从透明渐变显现）
- b 也在第 3 位附近（因为它正在从 +158dp 位移回弹到 0）
- 这就是用户描述的"b 右边靠在任务栏的右边沿"——此时的序列还没有变化，是原始顺序正在从拖拽状态"复位"

#### 阶段二：延迟 200ms 后（200ms 之后）

`scope.launch { delay(200) }` 中的代码开始执行（[HomeScreen.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt) 第 375-393 行）：

```kotlin
scope.launch {
    kotlinx.coroutines.delay(200)  // 延迟 200ms，等待位移动画基本结束
    if (finalIsDragOver && finalDraggingPlanId != null && finalInsertIndex >= 0) {
        // ... 计算新顺序 ...
        viewModel.reorderPinnedPlans(newOrder.map { it.id })  // ← 终于更新数据了！
    }
}
```

`viewModel.reorderPinnedPlans()` 被调用 → 数据库更新 → `pinnedPlans` Flow 发出新列表 `[c, a, b, d]` → LazyRow 收到新数据后重新组合布局 → 卡片位置从 `a,b,c,d` 跳转到 `c,a,b,d`。

此时上一阶段的 spring 位移动画已经跑完了，卡片已经"回到原位"了。现在又因为数据变化直接跳到新位置，产生了一次"硬切换"式的视觉跳跃。

#### 问题本质

用 `delay(200)` 错开两个动画（位移归零 vs 数据重排）虽然避免了它们叠加晃动，但也制造了另一个问题：**中间这 200ms 显示的是"已经复位的原始顺序"，用户能看到这个状态，然后再跳到最终顺序。** 中间态被暴露给用户了，形成了视觉断层。

用户本应看到的理想时序是：`a,b,c,d`（拖拽前）→ 拖拽中（c 浮动，a、b 让位）→ **松手后直接平滑过渡到** `c,a,b,d`（最终）

实际用户看到的时序是：`a,b,c,d`（拖拽前）→ 拖拽中（c 浮动，a、b 让位）→ 松手 → **a,b,c,d 复位**（中间态，可见 200ms）→ **跳到 c,a,b,d**（最终）

多出来的"a,b,c,d 复位"这一帧，就是造成"先看到 b 在第三位再弹回 c"的根本原因。

### 解决方案方向（待实施，当前版本暂不修改）

核心思路：**不要暴露"数据已复位"的中间态**，让用户从拖拽状态直接过渡到最终顺序。

可选方案：

**方案 A：不延迟数据更新，用动画承接**
- 松手时立刻调用 `viewModel.reorderPinnedPlans()` 更新数据，不做任何 delay
- 利用 `animateItemPlacement`（需要 `androidx.compose.animation:animation` 依赖）让 LazyRow 的 item 在数据变化时自动平滑动画到新位置
- 同时 `isDraggingFromTaskBar` 可以稍晚一点清除（比如 delay 50ms），让位移 spring 的回弹和 animateItemPlacement 的位移动画融合

**方案 B：使用本地状态先重排，不显示复位态**
- 松手时不立即清除 `isDraggingFromTaskBar` 和 `draggingPlanId`
- 立刻用 `pinnedPlans` + 目标位置算出新顺序，存入 `localPinnedPlans` 本地状态
- LazyRow 在拖拽结束的过渡期间使用 `localPinnedPlans` 作为数据源（此时卡片已经是新顺序了）
- 同时调用 ViewModel 同步到数据库
- 等 ViewModel 的 Flow 数据更新后，再切回 `pinnedPlans` 并清除拖拽状态

**方案 C：在被拖卡片上保持浮层，直到数据更新完成**
- 松手时不立即隐藏拖拽预览浮层（DragAndDropPreview），让它"着陆"到目标位置
- 在预览浮层下方，卡片数据已经悄悄更新为新顺序
- 预览浮层用一个 150~200ms 的动画（alpha 淡出 + 位置精确对齐到目标卡片位置）做"着陆"
- 动画结束后再隐藏浮层，此时下层真实卡片已经在正确位置，用户感知不到断层

## 问题四十：任务栏选中卡片松手后仍偏移，其他卡片正常

### 问题现象

任务栏中有 4 个策划卡片，原始顺序为 a（20260617-1）、b（20260617-2）、c（20260617-3）、d（20260617-4），从左到右排列。长按第 3 位的 c 卡片，拖拽到第 1 位的位置后松手。观察到：

1. 松手后，c 卡片（被选中拖拽的那张）仍然处于偏移状态，覆盖在 d 卡片上方
2. 其他卡片（a、b、d）位置正常，没有偏移残留
3. 只有**被长按选中并拖拽的那张卡片**出现偏移残留，其他卡片均正常

### 问题分类

第 5 类 —— 业务逻辑问题（拖拽排序状态一致性保障）

### 触发方式（精确复现步骤）

1. 首页任务栏中有 3 个或更多策划卡片（a、b、c、d ...）
2. 手指长按任意一张卡片（如第 3 位的 c），直到震动反馈表明进入拖拽模式
3. 保持手指不松手，将卡片拖拽到新的位置（如第 1 位，a 的左边）
4. 确保绿色目标位置指示框出现在预期的新位置
5. 松开手指结束拖拽
6. 观察：被拖拽的那张卡片（c）松手后仍处于偏移状态，没有回到正常位置，其他卡片位置正常

### 问题本质原因（四层竞态分析）

这是一个典型的**状态一致性竞态问题**，核心根源是数据库更新的非原子性与 Compose 声明式 UI 系统的异步性叠加，形成了四层竞态窗口，最终导致被拖拽卡片的 Animatable 状态未能正确重置。

#### 第一层：多次独立 UPDATE 引发多次重组（根源）

**代码位置**：
- [HomeViewModel.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeViewModel.kt#L75-L81) 第 75-81 行 `reorderPinnedPlans()` 函数
- [PlanDao.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/data/local/dao/PlanDao.kt#L38-L39) 第 38-39 行 `updatePinnedOrder()` 方法

```kotlin

// PlanDao.kt:38-39
```

`reorderPinnedPlans()` 采用 `forEachIndexed` 循环逐行调用 `updatePinnedOrder()`，每次调用都是一次**独立的** Room 数据库 UPDATE 操作。假设有 4 个卡片需要重新排序（a、b、c、d → c、a、b、d），就会产生 4 次独立的 UPDATE。

Room 的 Flow 观察者机制（[PlanDao.kt:18](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/data/local/dao/PlanDao.kt#L18) `getPinnedPlans()`）会在**每次**数据库变更时发射新的列表数据。因此 4 次 UPDATE 会导致 `pinnedPlans` Flow 发射 **4 次**，触发 LazyRow **4 次重新组合**。

这是整个问题的**根源**——本该是一次原子事务的排序操作，被拆成了多次独立更新，制造了多次重组的机会窗口。

#### 第三层：LazyRow 节点复用导致 Animatable 状态与新参数不同步


`LazyRow` 使用 `key = { it.id }` 进行节点复用。当卡片顺序变化时（如 c 从索引 2 移到索引 0），Compose 会将 c 卡片对应的**节点**从索引 2 的位置移动到索引 0 的位置，而不是销毁重建。

`remember { Animatable(0f) }` 的状态是与**节点**绑定的，不是与参数绑定的。节点复用意味着 Animatable 的内部值（`shiftAnim.value`）会被完整保留。

在多次重组的竞态窗口中：
1. 第一次重组：c 卡片的节点从索引 2 移动到索引 0，`shiftAnim.value` 可能还保留着拖拽中的偏移值（如 +158dp）
2. 此时如果 `isDragActive` 被读取为 `true`（第二层竞态），`graphicsLayer` 就会直接使用这个旧的偏移值

#### 第四层：LaunchedEffect 动画协程与 graphicsLayer 读取的时序差

**代码位置**：
- [HomeScreen.kt:884-890](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt#L884-L890) 第 884-890 行 `LaunchedEffect(isDragActive)` 重置逻辑


这里有一个关键的**时序差**：
- `graphicsLayer` 的 lambda 是在**组合阶段同步执行**的，每次重组都会立即读取 `shiftAnim.value`
- `LaunchedEffect` 中的 `snapTo(0f)` 是一个 **suspend 函数**，需要经过协程调度器的调度才能执行

竞态时序如下（以被拖拽的 c 卡片为例）：

| 时间点 | 操作 | 结果 |
|--------|------|------|
| T0 | 松手，`handleTaskBarDragEnd()` 执行 | `isDraggingFromTaskBar = false`，但快照尚未传播 |
| T1 | 第一次 UPDATE 完成，Flow 发射，触发第一次重组 | 开始组合 |
| T2 | `graphicsLayer` 执行，读取 `isDragActive` | 竞态：可能读到 `true`（快照未传播） |
| T3 | `graphicsLayer` 继续执行，读取 `shiftAnim.value` | 读到旧的偏移值（如 +158dp），因为 `snapTo` 还没执行 |
| T4 | `LaunchedEffect(isDragActive)` 协程被调度 | 开始执行 |
| T5 | `shiftAnim.snapTo(0f)` 执行 | Animatable 重置为 0，但画面已经在 T3 绘制完成 |
| T6 | 第二次 UPDATE 完成，触发第二次重组 | 理论上应该修正，但如果竞态条件持续存在，可能重复 T2-T5 |

这就是为什么**只有被拖拽的卡片**出现问题：
- 被拖拽的卡片在拖拽过程中 `shiftAnim.value` 被设置了非零的偏移值（+158dp 或 -158dp）
- 其他卡片的 `shiftAnim.value` 始终是 0，即使竞态发生，读取到的也是 0，视觉上没有异常
- 这完美解释了"其他卡片正常，只有选中的卡片偏移"的现象

### 之前修改失效的原因

之前在 `graphicsLayer` 中增加的硬编码条件（第 921 行 `!isDragActive -> 0f`）和新增的 `LaunchedEffect(isDragActive)` 重置逻辑（第 884-890 行），**只有在竞态条件不触发时才有效**：

1. `graphicsLayer` 的硬编码守卫依赖于 `isDragActive` 被**正确读取为 `false`**，但在第二层竞态中它可能被读取为 `true`
2. `LaunchedEffect` 的 `snapTo(0f)` 是**异步**的，赶不上 `graphicsLayer` 的**同步**读取
3. 这两个修改都没有解决根源问题——**多次独立 UPDATE 导致的多次重组**，只是在下游试图"兜底"，而竞态窗口总是能绕过兜底逻辑

## 问题三十九：图片选择器单击图片立即返回编辑页，无法多选

### 问题现象

新建策划页面点击"从相册中选择图片"，打开系统相册后，单击任意一张图片，相册立即关闭并返回到编辑策划界面，只能选择一张图片。只有长按第一张图片才能进入多选模式。用户期望的是默认进入多选模式，点击图片就是勾选，点击确认按钮才返回。

### 问题原因

**代码位置**：[CreatePlanScreen.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/create/CreatePlanScreen.kt)

先后尝试了两种方案均未解决根本问题：

#### 方案一（v1.2.4~v1.2.5）：`PickMultipleVisualMedia` + `OpenMultipleDocuments`

```kotlin
// Android 13+ 使用 PickMultipleVisualMedia
val photoPickerLauncher = rememberLauncherForActivityResult(
    contract = ActivityResultContracts.PickMultipleVisualMedia(),
    onResult = { uris -> viewModel.addUris(uris) }
)

// Android <13 使用 OpenMultipleDocuments
val documentPickerLauncher = rememberLauncherForActivityResult(
    contract = ActivityResultContracts.OpenMultipleDocuments(),
    onResult = { uris -> viewModel.addUris(uris) }
)
```

**原因**：Android 13 (API 33) 的 Photo Picker（`PickMultipleVisualMedia`）默认行为就是**单击图片 = 选中并立即返回**。虽然 API 名字叫 "PickMultiple"，但进入多选模式的前提是用户长按第一张图片。这是系统级别的行为，不是应用代码能控制的。国产ROM（小米/华为/OPPO/vivo）的系统相册普遍遵循此行为。

#### 方案二（中间版本）：`ACTION_GET_CONTENT` + `EXTRA_ALLOW_MULTIPLE`

```kotlin
val intent = Intent(Intent.ACTION_GET_CONTENT).apply {
    type = "image/*"
    putExtra(Intent.EXTRA_ALLOW_MULTIPLE, true)
    addCategory(Intent.CATEGORY_OPENABLE)
}
```

**原因**：`EXTRA_ALLOW_MULTIPLE` 在 Android 上只是一个 **hint（建议）**，不是强制要求。实际的多选行为完全由被调用的第三方相册应用决定。绝大多数国产ROM相册忽略这个 flag，保持"单击即选一张并关闭"的默认行为。

### 触发方式

1. 首页点击右下角 + 按钮 → 进入新建策划页面
2. 点击"从相册中选择图片"区域
3. 系统相册/文档选择器打开
4. 单击任意一张图片
5. 预期：图片被勾选，继续选择其他图片
6. 实际：相册立即关闭，返回编辑页，只选中了这一张图片

### 解决方案

**最终方案**：改用 `ActivityResultContracts.OpenMultipleDocuments()`（SAF 存储访问框架文档选择器）

```kotlin
val photoPickerLauncher = rememberLauncherForActivityResult(
    contract = ActivityResultContracts.OpenMultipleDocuments(),
    onResult = { uris -> viewModel.addUris(uris) }
)

val launchImagePicker: () -> Unit = {
    photoPickerLauncher.launch(arrayOf("image/*"))
}
```

**为什么 SAF 可以解决问题**：

1. **行为一致性**：SAF（Storage Access Framework）是 Android 4.4 (API 19) 引入的标准框架，`OpenMultipleDocuments` 在所有 Android 5.0+ 设备上行为一致
2. **默认多选**：SAF 的文档选择器默认就是多选模式，点击文件 = 勾选（不会立即关闭），必须点击"打开/确认"按钮才返回所选列表
3. **系统级保证**：SAF 由系统 DocumentsUI 提供标准界面，不受第三方相册实现影响
4. **权限持久化**：通过 `takePersistableUriPermission` 可以获取持久化访问权限（虽然本项目暂时不需要）

**同步清理**：移除不再需要的 import（`Intent`、`Build`、`PickVisualMediaRequest` 等）

**提示文字更新**：从"可长按多选"改为"点击多选"

### 解决效果

- 所有 Android 版本、所有厂商 ROM 上行为一致
- 点击图片 = 勾选/取消勾选，不会立即返回
- 底部有"打开"或"完成"按钮，点击后才返回并返回所有选中的图片
- 用户可以从容选择多张图片，无需长按

## 问题四十：任务栏拖拽卡片无法拖到第1位，拖到第2位自动结束

### 问题现象

任务栏中有 4 个策划卡片 a、b、c、d（从左到右）。长按住最右侧的 d 卡片向左拖动排序，目标是放到 a 的前面（第 0 位，最左侧）。但实际拖拽时：
- 可以拖过 c，到达 b 原来的位置（第 2 位 → 第 1 位，即 b 和 c 之间）
- 但无法继续向左移动到 a 的前面
- **即使手指没有松开，拖拽也会自动结束**，d 停留在第 2 位（b 的位置）

这是严重的逻辑问题——拖拽过程未完成就提前终止，且无法到达第一个位置。

### 问题原因

**涉及文件**：[HomeScreen.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt)

问题由三个独立 Bug 叠加导致：

#### Bug A：targetInsertIndex 视口外估算算法错误（核心逻辑Bug）

**代码位置**：`targetInsertIndex` 的 `derivedStateOf` 中视口左侧估算部分

```kotlin
// 原算法（有Bug）
dropXInContainer < viewportLeft -> {
    var idx = firstVisibleIndex
    while (idx > 0) {
        val itemCenterEstimate = (idx - 1) * totalItemWidthPx + itemWidthPx / 2f
        if (dropXInContainer < itemCenterEstimate) {
            idx--
        } else {
            break  // ← 致命错误：一旦某一步不满足就停止循环！
        }
    }
    idx
}
```

**原因**：假设序列 `a(0), b(1), c(2), d(3)`，视口可见 `b(1), c(2), d(3)`，a 已滚出左侧。

当手指拖到任务栏左侧但仍在视口内边缘时：
- `firstVisibleIndex = 1`（b 的 index）
- `dropXInContainer` 大于 `idx-1=0`（a）的估算中心（因为还没拖到最左边）
- 第一次 while 判断：`dropXInContainer > itemCenterEstimate(0)` → 进入 `else` 分支 → **break**
- 循环结束，`idx` 停在 1，**永远到不了 0**

循环的条件判断方向反了：应该是"当 dropX 小于第 idx-1 项的中心时，idx 减1"，但实际上应该从 `firstVisibleIndex` 一直循环到 0，而不是遇到一个不满足就 break。

#### Bug B：自动滚动触发范围过小 + 速度不足

**代码位置**：`LaunchedEffect(isDraggingFromTaskBar, draggingPlanId)` 自动滚动块

**原因**：
1. `edgeZone = 80dp`：触发区域太小，手指必须紧贴边缘才触发滚动
2. `maxScrollStep = 8dp`：每帧滚动步长太小，滚动缓慢
3. Y轴范围仅限任务栏内部：手指稍微移出任务栏上下方就停止滚动
4. `touchSlop` 和滚动触发的配合问题：用户向左拖动时，水平移动距离超过 touchSlop 后就应该触发向左滚动，但原逻辑需要手指进入边缘区才滚

#### Bug C：手势检测在 LazyRow 滚动时意外取消

**代码位置**：每个 PinnedPlanItem 内部的 `detectDragGesturesAfterLongPress`

**原因**：原实现是在每个 `PinnedPlanItem` 组件上**独立检测**长按和拖动手势。当自动滚动触发时，LazyRow 的内容发生滚动，item 的位置在视口中变化，会导致 Compose 的手势检测系统认为手势被取消（`onDragCancel` 被调用）。`onDragCancel` 直接调用了 `onDragEnd()`，导致**手指还没松开，拖拽就已经结束了**。

这就是"即便没有松手，滑动就已经结束"的根本原因——不是用户松手了，而是手势检测系统因为滚动导致的坐标变化主动取消了手势。

### 触发方式（精确复现步骤）

1. 首页任务栏中有 4 个或更多策划卡片（a、b、c、d 从左到右）
2. 手指长按最右侧的 d 卡片，直到震动反馈
3. 保持手指不松开，向左缓慢拖动
4. 拖过 c，继续向左，目标是 a 的前面
5. 观察：拖到大约 b 的位置时，d 突然落下（拖拽结束），手指还在屏幕上
6. 观察：最终 d 停在第 2 位，无法到达第 1 位（a 前面）

### 解决方案

从三个层面同时修复：

#### 修复 A：targetInsertIndex 算法重写

```kotlin
val targetIdx: Int = when {
    dropXInContainer <= 0f -> {
        0  // 直接在最左端，返回第0位
    }
    dropXInContainer < viewportLeft -> {
        val distanceLeft = viewportLeft - dropXInContainer
        val stepsBack = (distanceLeft / totalItemWidthPx).toInt() + 2
        (firstVisibleIndex - stepsBack).coerceAtLeast(0)
    }
    dropXInContainer > viewportRight -> {
        val distanceRight = dropXInContainer - viewportRight
        val stepsForward = (distanceRight / totalItemWidthPx).toInt() + 2
        (lastVisibleIndex + stepsForward).coerceAtMost(pinnedPlans.size)
    }
    else -> {
        // 视口内保持原有的遍历逻辑
    }
}
```

关键改动：
1. `dropXInContainer <= 0` 直接返回 0，不经过任何循环判断
2. 视口外改用**距离/卡片宽度**数学估算，不用 while+break 的方式
3. 增加 2 个卡片的余量步长，确保估算有富余
4. 用 `coerceAtLeast(0)` 和 `coerceAtMost(pinnedPlans.size)` 强制边界

#### 修复 B：自动滚动参数增强

```kotlin
val topBound = taskBarBounds.top - with(density) { 200.dp.toPx() }
val bottomBound = taskBarBounds.bottom + with(density) { 200.dp.toPx() }
val inYRange = dragPosition.y >= topBound && dragPosition.y <= bottomBound
val edgeZone = with(density) { 120.dp.toPx() }  // 80dp → 120dp
val maxScrollStep = with(density) { 14.dp.toPx() } // 8dp → 14dp
```

关键改动：
1. Y 轴检测范围扩大到任务栏上下各 200dp，手指移出任务栏仍然继续滚动
2. 边缘触发区从 80dp → 120dp，更早触发滚动
3. 最大滚动步长从 8dp → 14dp，滚动更快
4. 速度因子调整为 0.3 + 0.7 × speedFactor，边缘越远滚得越快

#### 修复 C：全局手势检测 + 事件前置消费

**重构架构**：将长按+拖拽检测从每个 PinnedPlanItem 提升到**根 Box 层级**，使用 `awaitEachGesture` + `awaitFirstDown` + `awaitPointerEvent` 手动控制完整手势流程。

```kotlin
Box(modifier = Modifier.fillMaxSize()
    .pointerInput(Unit) {
        awaitEachGesture {
            val down = awaitFirstDown(requireUnconsumed = false)
            
            // 长按检测
            var longPressTriggered = false
            val longPressJob = scope.launch {
                delay(longPressTimeout)
                if (!hasMoved && pressedPlan != null) {
                    longPressTriggered = true
                    // ... 触发拖拽
                }
            }
            
            // 全程捕获事件
            do {
                val event = awaitPointerEvent(PointerEventPass.Initial)
                val change = event.changes.firstOrNull { it.id.value == down.id.value }
                if (longPressTriggered && draggingPlanId != null) {
                    change.consume()  // ← 关键：拖拽开始后消费所有事件
                    // 更新位置
                }
            } while (!event.changes.all { it.changedToUp() })
            
            // 结束处理
        }
    }
)
```

关键改动：
1. **全局单一手势检测**：整个屏幕只用一个 pointerInput，在根 Box 上检测所有触摸
2. **`PointerEventPass.Initial`**：在所有子组件（包括 LazyRow）之前获取事件
3. **拖拽开始后 `consume()` 所有事件**：确保 LazyRow 和 PinnedPlanItem 收不到事件，不会触发滚动冲突或手势取消
4. **按指针 ID 追踪**：用 `down.id.value` 匹配事件，防止多指触摸时丢失追踪
5. **`requireUnconsumed = false`**：即使事件已经被消费也能收到，确保不会漏掉 down 事件
6. **`Throwable` 兜底捕获**：防止任何异常导致手势意外终止

同时，PinnedPlanItem 移除内部的 `detectDragGesturesAfterLongPress`，在拖拽激活时（`isDragActive=true`）也禁用 click 事件。

#### 修复 D：拖拽时禁用 LazyRow 用户滚动

```kotlin
LazyRow(
    state = taskBarListState,
    userScrollEnabled = isUserScrollEnabled  // 拖拽时 = false
)
```

长按触发拖拽的同时设置 `isUserScrollEnabled = false`，防止用户拖拽过程中 LazyRow 的滚动手势与拖拽冲突。

### 解决效果

- 任务栏卡片可以顺畅地从任意位置拖拽到任意位置（包括最左端和最右端）
- 拖拽过程中手指不松开，手势就不会意外终止
- 拖到边缘时自动平滑滚动，滚动速度与越界距离成正比
- targetInsertIndex 可以正确到达 0 和 pinnedPlans.size 的边界位置

## 问题四十一：构建失败 - awaitEachGesture 受限作用域内无法调用 launch/delay

### 问题现象

推送代码到 GitHub 后，CI 构建失败，报错：

```
> Task :app:compileDebugKotlin FAILED

e: HomeScreen.kt:349:44 Unresolved reference. None of the following candidates is applicable because of receiver type mismatch:
   fun CoroutineScope.launch(context: CoroutineContext = ..., start: ..., block: suspend CoroutineScope.() -> Unit): Job

e: HomeScreen.kt:350:29 Restricted suspending functions can invoke member or extension suspending functions only on their restricted coroutine scope.

e: HomeScreen.kt:350:29 Suspension functions can only be called within coroutine body.
```

### 问题原因

**代码位置**：[HomeScreen.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt#L349-L350)

**问题分类**：第4条 - 编译/语法问题

**根本原因**：

在 Compose 的 `pointerInput { ... }` 中调用了 `awaitEachGesture { ... }`，而 `awaitEachGesture` 的 lambda 参数是一个 **restricted suspending function（受限制的挂起函数）**。

受限制的挂起函数有严格的调用限制：
1. **不能直接调用 `CoroutineScope.launch`** - 因为 `launch` 是 `CoroutineScope` 的扩展函数，而受限作用域不是 `CoroutineScope`
2. **不能直接调用普通挂起函数（如 `delay`）** - 受限挂起函数内只能调用该受限作用域自己的成员或扩展挂起函数
3. **只能调用其受限协程作用域的成员/扩展挂起函数**

原代码在 `awaitEachGesture` 的 lambda 内部直接写了 `launch { delay(...) }`，违反了上述限制。

### 触发方式

在 `awaitEachGesture` 的受限挂起函数作用域内直接调用 `launch { }` 和 `delay()`。

### 解决方案

代码中已经在外部通过 `rememberCoroutineScope()` 定义了 `scope` 变量。使用 `scope.launch { }` 而不是直接 `launch { }`。

修改前：
```kotlin
awaitEachGesture {
    val longPressJob = launch {   // 错误：受限作用域内没有 CoroutineScope
        delay(longPressTimeout)   // 错误：受限作用域内不能直接调用 delay
        ...
    }
}
```

修改后：
```kotlin
awaitEachGesture {
    val longPressJob = scope.launch {   // 正确：使用外部 rememberCoroutineScope() 的 scope
        delay(longPressTimeout)          // 正确：在 scope.launch 的协程体内
        ...
    }
}
```

**原理**：`scope` 是通过 `rememberCoroutineScope()` 创建的，它绑定到当前 Composable 的生命周期，是一个标准的 `CoroutineScope`。在受限挂起函数内可以访问外部定义的 `scope` 变量并调用其 `launch` 方法，因为这是在调用一个对象（scope）的方法，而不是在受限作用域内直接启动协程。

### 解决效果

- CI 编译通过
- 长按协程正常启动和取消
- 功能行为与设计预期一致

5. 观察：高亮卡片不会回归原位，仍然以偏移状态悬浮，与其他卡片重叠

### 问题原因

**涉及文件**：[HomeScreen.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt)

**问题分类**：第 3 类（Bug 原因与触发方式）+ 第 5 类（业务逻辑问题）

这是一个 Compose 状态管理与 LazyRow 组件复用的**复合问题**，由以下多个层次的 Bug 叠加导致，前几次修复只触及了表层，未能解决根因：

---

#### 层次一：LazyRow key 复用 + remember Animatable 状态残留（根因）

**代码位置**：`LazyRow(items(pinnedPlans, key = { it.id }) { plan -> ... PinnedPlanItem(...) }`，以及 `PinnedPlanItem` 内部：

```kotlin
val shiftAnim = remember { Animatable(0f) }
val alphaAnim = remember { Animatable(1f) }
val scaleAnim = remember { Animatable(1f) }
```

`LazyRow` 使用 `key = { it.id }` 进行组件复用——同一个 `plan.id` 的卡片在数据重新排序后会被 Compose 视为**同一个 Composable 实例**，其 `remember` 的所有状态（包括三个 Animatable）都会被**完整保留**，不会重置。

但问题是：**Animatable 的值与它的"逻辑位置"绑定，而不是与 plan.id 绑定**。举例说明：

| 时刻 | pinnedPlans 顺序 | 被拖拽卡片 | shiftAnim 状态（plan.id=X 的卡片） |
|------|------------------|-----------|------------------------------------|
| 拖拽前 | [A, B, C] | — | shiftAnim(0f) |
| 拖拽中（拖到第 0 位） | [A, B, C] | B | B自己 alpha→0f；A 的 shiftAnim→+158dp（右移让位） |
| 松手瞬间（isDraggingFromTaskBar=false，pinnedPlans 还未更新） | [A, B, C] | B | A、B、C 全部 snapTo(0f) / alpha=1f |
| 数据库更新后 pinnedPlans 变为 [B, A, C] | [B, A, C] | — | **问题**：B 卡片（原 index=1，现 index=0）被 LazyRow 复用，其 Animatable 可能保留了它作为"被拖拽卡片"时的 alpha=0f→1f 动画中间态或位移状态残留 |

当 `pinnedPlans` 从 `[A, B, C]` 变为 `[B, A, C]` 时：
- key=B 的 Composable 从 index=1 移到了 index=0
- 它在 index=1 时的 Animatable 状态（拖拽中 alpha=0f、或位移 snapTo 的中间值）被 remember 保留
- 在新位置 index=0 继续使用这些残留值
- 造成视觉上的"偏移不归位"

这就是为什么 `snapTo(0f)` 和 `safeTranslationX = if (isDragActive) shiftAnim.value else 0f` 都不能完全解决问题：**snapTo 是在协程中异步执行的，而 pinnedPlans 更新触发的重组可能发生在 snapTo 执行完成之前**；safeTranslationX 虽然在 !isDragActive 时强制为 0f，但 `isDraggingFromTaskBar` 的变化与 `pinnedPlans` 的变化之间存在竞态窗口，期间卡片被复用时 remember 的状态仍是脏的。

---

#### 层次二：highlighted 边框的状态依赖 isDraggingFromTaskBar + index 计算，竞态导致残留

**代码位置**：LazyRow items lambda 内：

```kotlin
val isNearTargetSlot = isDraggingFromTaskBar &&
    !isThisPinnedDragging &&
    targetInsertIndex >= 0 &&
    (index == targetInsertIndex || index == targetInsertIndex - 1)
// ...
PinnedPlanItem(
    highlighted = isNearTargetSlot,
    isDragActive = isDraggingFromTaskBar
)
```

在 `handleTaskBarDragEnd` 中：

```kotlin
draggingPlanId = null
isDraggingFromTaskBar = false   // ← 先设置状态
isDragOver = false
// ...
scope.launch {
    // ...
    viewModel.reorderPinnedPlans(newOrder.map { it.id })  // ← 后更新数据（异步）
}
```

时序问题：
1. `isDraggingFromTaskBar = false` 立即触发一次重组 → 此时 `pinnedPlans` 还是旧顺序，所有 `highlighted` 应变为 false
2. 数据库 Flow 更新 `pinnedPlans` → 触发第二次重组 → 此时 LazyRow 按新顺序重新分发 item，items lambda 重新计算每个 plan 的 `index`
3. **关键竞态**：如果第二次重组时 Compose 的 Snapshot 系统还未完全同步 `isDraggingFromTaskBar` 的最新值（或者 `derivedStateOf` 的 `targetInsertIndex` 还保留着旧值），就会在新顺序的某个卡片上错误地计算出 `isNearTargetSlot=true`，导致绿色边框残留

更严重的是：`targetInsertIndex` 是 `derivedStateOf`，它的更新依赖于 `dragPosition`、`taskBarListState` 等多个状态，**在 `dragPosition` 被设为 Offset.Zero 后，derivedStateOf 的重新计算是异步的**，不是瞬时完成的。如果 pinnedPlans 更新先于 targetInsertIndex 归零，就会在新顺序上用旧的 targetInsertIndex 计算出错误的 highlighted。

---

#### 层次三：全局浮层消失与卡片状态恢复不同步（视觉残留的直接原因）

全局拖拽预览卡片：
```kotlin
val draggingPlan = plans.find { it.id == draggingPlanId } ?: pinnedPlans.find { it.id == draggingPlanId }
if (draggingPlan != null) {  // ← draggingPlanId=null 时立即消失
    Box(modifier = Modifier.zIndex(1000f)) { ... }
}
```

松手后 `draggingPlanId = null` 先被设置 → 全局浮层立即从组合中移除。但此时 LazyRow 中对应 plan.id 的卡片的 Animatable（alpha 从 0f → 1f）还在动画过程中，或者因为复用问题 alpha 还是 0 → 用户看到的是"全局卡片消失了，但下面那张原始卡片没有以正常状态出现，而是以偏移/半透明状态叠在那里"。

### 为什么前几次修改没有解决问题

| 修复尝试 | 做了什么 | 为什么无效 |
|---------|----------|-----------|
| 第 1 次（v1.2.7 第一轮） | 在 LaunchedEffect(isDragActive) 中调用 snapTo(0f/1f) | `snapTo` 是 suspend 函数，需要在协程中调度执行。但 `pinnedPlans` 数据更新触发的重组可能在 snapTo 执行之前就发生了，此时 Animatable 的值还是旧的。而且 LazyRow key 复用导致 remember 的状态跨排序保留。 |
| 第 2 次（v1.2.7 第二轮） | 增加 safeTranslationX / safeAlpha / safeScale，!isDragActive 时直接返回 0f/1f，不读取 Animatable | 这层保护了 graphicsLayer 不受 Animatable 旧值影响，但 **highlighted 边框的计算在 LazyRow items lambda 中**，它依赖于 `isDraggingFromTaskBar` 和 `targetInsertIndex` 的快照一致性。如果 targetInsertIndex 在 pinnedPlans 更新后还没归零，highlighted 仍为 true，且 `isDragActive`（即 isDraggingFromTaskBar）在竞态窗口内可能还是 true，绿色边框就会残留。另外，safeTranslationX 虽然强制为 0f，但 Compose 的重组是批量的，若与 items lambda 的 index 计算不同步，仍可能出现短暂错位。 |
| 第 2 次同时 | 移除 handleTaskBarDragEnd 中的 delay(50) | 消除了 50ms 的延迟确实减少了暴露中间态的窗口，但 viewMode.reorderPinnedPlans 本身是异步的（数据库写入 + Flow 分发），仍与 Compose 状态变化存在竞态。 |

**根本未被触及的问题**：
1. **Animatable 的 remember 没有绑定 plan.id + index**，导致 LazyRow 复用时状态残留
2. **pinnedPlans 数据更新与 isDraggingFromTaskBar 状态变化是两个独立的异步流程**，没有做同步协调
3. **targetInsertIndex 的 derivedStateOf 归零时机晚于 pinnedPlans 更新时机**，导致用旧的 targetInsertIndex 在新顺序上计算出错误的 highlighted

### 正确的修复方向（待实施，当前版本暂不修改代码）

需要从架构层面解决，单点 patch 无法根治：

**方案 1：Animatable 的 remember key 绑定 plan.id + index**
```kotlin
val shiftAnim = remember(plan.id, index) { Animatable(0f) }
val alphaAnim = remember(plan.id, index) { Animatable(1f) }
val scaleAnim = remember(plan.id, index) { Animatable(1f) }
```
每当 plan.id 或 index 变化时，Animatable 重新初始化为初始值，不复用旧值。

**方案 2：拖拽结束时"先更新数据，再清除状态"，引入本地过渡状态**
- 松手时不立即清除 `isDraggingFromTaskBar` 和 `draggingPlanId`
- 先计算新顺序，更新到一个**本地 MutableState** `localPinnedPlans`，让 LazyRow 立即使用新顺序渲染
- 同时调用 ViewModel 同步数据库
- 等 ViewModel 的 Flow 返回的数据与 `localPinnedPlans` 一致时，再清除拖拽状态
- 这样不会出现"旧顺序复位 → 新顺序跳转"的断层

**方案 3：highlighted 强制依赖 !isDraggingFromTaskBar → false 的单向信号**
增加一个 `dragJustEnded` MutableState，拖拽结束时设为 true 并延迟 200ms 后设为 false，PinnedPlanItem 的 `highlighted = isNearTargetSlot && !dragJustEnded`，确保松手后边框立即清除，不参与竞态。

**方案 4（最彻底）：不使用 Animatable + remember，全部改用 derivedStateOf 计算**
```kotlin
val translationX by remember(isDragActive, shiftOffsetPx) {
    derivedStateOf { if (isDragActive) shiftOffsetPx else 0f }
}
```
不使用 Animatable（它内部有可变状态且是 suspend），直接用 derivedStateOf 计算。位移动画改用 `animate*AsState`（Compose 会自动管理生命周期，无需手动 snapTo）。
