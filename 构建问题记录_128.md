## 问题四十二：任务栏卡片拖拽松手后仍不归位、悬浮在其他卡片上方（多次修复仍存在）

### 问题现象

在任务栏内长按任意策划卡片（如 20260617-3），向左或向右拖动排序，松手后观察：

1. 被拖拽的目标卡片仍然以"高亮绿色边框"（highlighted）的视觉状态，偏移覆盖在另一张卡片的上方
2. 卡片没有回归到 LazyRow 正常布局的位置，而是悬浮显示
3. 全局的拖拽预览浮层（zIndex=1000 的卡片）已经消失，但任务栏内部的 LazyRow 卡片仍处于拖拽中的位移/高亮/半透明状态
4. 多次修复（添加 snapTo、safeTranslationX 安全变量、移除延迟等）后，该现象仍可稳定复现

### 触发方式（精确复现步骤）

1. 首页任务栏中有 3 个或更多策划卡片
2. 手指长按中间某张卡片（如第 2 张），直到震动反馈
3. 保持按住，向左或向右拖动一个卡片宽度以上，使其他卡片出现高亮绿色边框（目标位置指示）
4. 松开手指
5. 观察：高亮卡片不会回归原位，仍然以偏移状态悬浮，与其他卡片重叠

### 问题原因

**涉及文件**：[HomeScreen.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt)

**问题分类**：第 3 类（Bug 原因与触发方式）+ 第 5 类（业务逻辑问题）

这是一个 Compose 状态管理与 LazyRow 组件复用的**复合问题**，由以下多个层次的 Bug 叠加导致，前几次修复只触及了表层，未能解决根因：

---

#### 层次一：LazyRow key 复用 + remember Animatable 状态残留（根因）

**代码位置**：`LazyRow(items(pinnedPlans, key = { it.id }) { plan -> ... PinnedPlanItem(...) }`，以及 `PinnedPlanItem` 内部：

```kotlin
val shiftAnim = remember { Animatable(0f) }
val alphaAnim = remember { Animatable(1f) }
val scaleAnim = remember { Animatable(1f) }
```

`LazyRow` 使用 `key = { it.id }` 进行组件复用——同一个 `plan.id` 的卡片在数据重新排序后会被 Compose 视为**同一个 Composable 实例**，其 `remember` 的所有状态（包括三个 Animatable）都会被**完整保留**，不会重置。

但问题是：**Animatable 的值与它的"逻辑位置"绑定，而不是与 plan.id 绑定**。举例说明：

| 时刻 | pinnedPlans 顺序 | 被拖拽卡片 | shiftAnim 状态（plan.id=X 的卡片） |
|------|------------------|-----------|------------------------------------|
| 拖拽前 | [A, B, C] | — | shiftAnim(0f) |
| 拖拽中（拖到第 0 位） | [A, B, C] | B | B自己 alpha→0f；A 的 shiftAnim→+158dp（右移让位） |
| 松手瞬间（isDraggingFromTaskBar=false，pinnedPlans 还未更新） | [A, B, C] | B | A、B、C 全部 snapTo(0f) / alpha=1f |
| 数据库更新后 pinnedPlans 变为 [B, A, C] | [B, A, C] | — | **问题**：B 卡片（原 index=1，现 index=0）被 LazyRow 复用，其 Animatable 可能保留了它作为"被拖拽卡片"时的 alpha=0f→1f 动画中间态或位移状态残留 |

当 `pinnedPlans` 从 `[A, B, C]` 变为 `[B, A, C]` 时：
- key=B 的 Composable 从 index=1 移到了 index=0
- 它在 index=1 时的 Animatable 状态（拖拽中 alpha=0f、或位移 snapTo 的中间值）被 remember 保留
- 在新位置 index=0 继续使用这些残留值
- 造成视觉上的"偏移不归位"

这就是为什么 `snapTo(0f)` 和 `safeTranslationX = if (isDragActive) shiftAnim.value else 0f` 都不能完全解决问题：**snapTo 是在协程中异步执行的，而 pinnedPlans 更新触发的重组可能发生在 snapTo 执行完成之前**；safeTranslationX 虽然在 !isDragActive 时强制为 0f，但 `isDraggingFromTaskBar` 的变化与 `pinnedPlans` 的变化之间存在竞态窗口，期间卡片被复用时 remember 的状态仍是脏的。

---

#### 层次二：highlighted 边框的状态依赖 isDraggingFromTaskBar + index 计算，竞态导致残留

**代码位置**：LazyRow items lambda 内：

```kotlin
val isNearTargetSlot = isDraggingFromTaskBar &&
    !isThisPinnedDragging &&
    targetInsertIndex >= 0 &&
    (index == targetInsertIndex || index == targetInsertIndex - 1)
// ...
PinnedPlanItem(
    highlighted = isNearTargetSlot,
    isDragActive = isDraggingFromTaskBar
)
```

在 `handleTaskBarDragEnd` 中：

```kotlin
draggingPlanId = null
isDraggingFromTaskBar = false   // ← 先设置状态
isDragOver = false
// ...
scope.launch {
    // ...
    viewModel.reorderPinnedPlans(newOrder.map { it.id })  // ← 后更新数据（异步）
}
```

时序问题：
1. `isDraggingFromTaskBar = false` 立即触发一次重组 → 此时 `pinnedPlans` 还是旧顺序，所有 `highlighted` 应变为 false
2. 数据库 Flow 更新 `pinnedPlans` → 触发第二次重组 → 此时 LazyRow 按新顺序重新分发 item，items lambda 重新计算每个 plan 的 `index`
3. **关键竞态**：如果第二次重组时 Compose 的 Snapshot 系统还未完全同步 `isDraggingFromTaskBar` 的最新值（或者 `derivedStateOf` 的 `targetInsertIndex` 还保留着旧值），就会在新顺序的某个卡片上错误地计算出 `isNearTargetSlot=true`，导致绿色边框残留

更严重的是：`targetInsertIndex` 是 `derivedStateOf`，它的更新依赖于 `dragPosition`、`taskBarListState` 等多个状态，**在 `dragPosition` 被设为 Offset.Zero 后，derivedStateOf 的重新计算是异步的**，不是瞬时完成的。如果 pinnedPlans 更新先于 targetInsertIndex 归零，就会在新顺序上用旧的 targetInsertIndex 计算出错误的 highlighted。

---

#### 层次三：全局浮层消失与卡片状态恢复不同步（视觉残留的直接原因）

全局拖拽预览卡片：
```kotlin
val draggingPlan = plans.find { it.id == draggingPlanId } ?: pinnedPlans.find { it.id == draggingPlanId }
if (draggingPlan != null) {  // ← draggingPlanId=null 时立即消失
    Box(modifier = Modifier.zIndex(1000f)) { ... }
}
```

松手后 `draggingPlanId = null` 先被设置 → 全局浮层立即从组合中移除。但此时 LazyRow 中对应 plan.id 的卡片的 Animatable（alpha 从 0f → 1f）还在动画过程中，或者因为复用问题 alpha 还是 0 → 用户看到的是"全局卡片消失了，但下面那张原始卡片没有以正常状态出现，而是以偏移/半透明状态叠在那里"。

### 为什么前几次修改没有解决问题

| 修复尝试 | 做了什么 | 为什么无效 |
|---------|----------|-----------|
| 第 1 次（v1.2.7 第一轮） | 在 LaunchedEffect(isDragActive) 中调用 snapTo(0f/1f) | `snapTo` 是 suspend 函数，需要在协程中调度执行。但 `pinnedPlans` 数据更新触发的重组可能在 snapTo 执行之前就发生了，此时 Animatable 的值还是旧的。而且 LazyRow key 复用导致 remember 的状态跨排序保留。 |
| 第 2 次（v1.2.7 第二轮） | 增加 safeTranslationX / safeAlpha / safeScale，!isDragActive 时直接返回 0f/1f，不读取 Animatable | 这层保护了 graphicsLayer 不受 Animatable 旧值影响，但 **highlighted 边框的计算在 LazyRow items lambda 中**，它依赖于 `isDraggingFromTaskBar` 和 `targetInsertIndex` 的快照一致性。如果 targetInsertIndex 在 pinnedPlans 更新后还没归零，highlighted 仍为 true，且 `isDragActive`（即 isDraggingFromTaskBar）在竞态窗口内可能还是 true，绿色边框就会残留。另外，safeTranslationX 虽然强制为 0f，但 Compose 的重组是批量的，若与 items lambda 的 index 计算不同步，仍可能出现短暂错位。 |
| 第 2 次同时 | 移除 handleTaskBarDragEnd 中的 delay(50) | 消除了 50ms 的延迟确实减少了暴露中间态的窗口，但 viewMode.reorderPinnedPlans 本身是异步的（数据库写入 + Flow 分发），仍与 Compose 状态变化存在竞态。 |

**根本未被触及的问题**：
1. **Animatable 的 remember 没有绑定 plan.id + index**，导致 LazyRow 复用时状态残留
2. **pinnedPlans 数据更新与 isDraggingFromTaskBar 状态变化是两个独立的异步流程**，没有做同步协调
3. **targetInsertIndex 的 derivedStateOf 归零时机晚于 pinnedPlans 更新时机**，导致用旧的 targetInsertIndex 在新顺序上计算出错误的 highlighted

### 正确的修复方向（v1.2.7 最终修复）

最终采用"方案 1 + 方案 3"组合修复（方案 4 animateFloatAsState 不可用）：

**方案 1：Animatable 的 remember key 绑定 plan.id + isDragActive + isDragging**
```kotlin
val pinnedShiftAnim = remember(key1 = plan.id, key2 = isDragActive, key3 = isDragging) { Animatable(0f) }
val pinnedAlphaAnim = remember(key1 = plan.id, key2 = isDragActive, key3 = isDragging) { Animatable(1f) }
val pinnedScaleAnim = remember(key1 = plan.id, key2 = isDragActive, key3 = isDragging) { Animatable(1f) }
```
每当 plan.id 或拖拽状态变化时，Animatable 重新初始化为初始值，不复用旧值。非拖拽状态立即 snapTo 正常值 + graphicsLayer 硬编码守卫兜底。

**方案 3：highlighted 增加 dragJustEnded 状态守卫**
增加一个 `dragJustEnded` MutableState，拖拽结束时设为 true 并延迟 200ms 后设为 false，PinnedPlanItem 的 `highlighted = isNearTargetSlot && !dragJustEnded`，确保松手后边框立即清除，不参与竞态。

## 问题四十四：全局手势检测（awaitEachGesture）架构导致手势冲突，draggingPlanId 无法正确清空（v1.2.8 修复）

### 问题现象

v1.2.7 版本（问题四十二、四十三修复后）用户反馈："之前的问题全都没有解决"，具体表现：

1. 长按任务栏卡片进行拖拽排序，松手后，被选中的卡片无法回到正常位置
2. 点击屏幕上的其他任意区域，之前选中的卡片会"飞"到该点击位置
3. 拖拽状态似乎一直处于激活状态，无法正常退出
4. 全局拖拽预览浮层（zIndex=1000）不消失，一直跟随手指或停留在最后位置

### 触发方式（精确复现步骤）

1. 首页任务栏中有 3 个或更多策划卡片
2. 手指长按任意卡片直到震动反馈
3. 拖动任意距离后松手
4. 观察：松手后拖拽状态没有清除，卡片不回归
5. 点击屏幕其他空白区域
6. 观察：之前被拖拽的卡片移动到了点击位置

### 问题原因

**涉及文件**：[HomeScreen.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt)

**问题分类**：第 3 类（Bug 原因与触发方式）+ 第 5 类（业务逻辑问题）

这是 v1.2.6~v1.2.7 采用的"全局手势检测架构"（根 Box 的 `awaitEachGesture` + `PointerEventPass.Initial` + `consume()`）的**系统性缺陷**，不是单点 Bug。

---

#### 层次一：根 Box 的 awaitEachGesture 与子组件手势冲突（核心根因）

**代码位置**：根 `Box(modifier = Modifier.fillMaxSize().pointerInput(Unit) { awaitEachGesture { ... } })`

v1.2.6 为了解决问题四十（拖拽无法到第 1 位、自动滚动时手势取消），将手势检测从每个 `PinnedPlanItem` 内部的 `detectDragGesturesAfterLongPress` 提升到**整个屏幕的根 Box 层级**，使用 `awaitEachGesture` 手动控制完整手势流程：

```kotlin
Box(modifier = Modifier.fillMaxSize()
    .pointerInput(Unit) {
        awaitEachGesture {
            val down = awaitFirstDown(requireUnconsumed = false)
            
            // 长按检测
            var longPressTriggered = false
            val longPressJob = scope.launch {
                delay(longPressTimeout)
                if (!hasMoved && pressedPlan != null) {
                    longPressTriggered = true
                    // ... 触发拖拽
                }
            }
            
            // 全程捕获事件
            do {
                val event = awaitPointerEvent(PointerEventPass.Initial)
                val change = event.changes.firstOrNull { it.id.value == down.id.value }
                if (longPressTriggered && draggingPlanId != null) {
                    change.consume()  // ← 消费所有事件
                    // 更新 dragPosition
                }
            } while (!event.changes.all { it.changedToUp() })
            
            // 结束处理：调用 handleTaskBarDragEnd()
        }
    }
)
```

**致命问题**：
1. **全局范围太大**：整个屏幕所有触摸事件都先经过根 Box，包括点击下方策划卡片、点击 + 按钮、点击返回键区域等
2. **事件消费逻辑不完整**：只有在 `longPressTriggered && draggingPlanId != null` 时才消费事件。但如果长按检测因任何原因（hasMoved=true、pressedPlan=null 等）未触发，`longPressTriggered` 会停留在 `false`，后续的 MOVE 和 UP 事件都不会被消费
3. **状态机不完整**：`awaitEachGesture` 中存在大量的可变状态（`longPressTriggered`、`hasMoved`、`pressedPlan`、`draggingPlanId`、`isDraggingFromTaskBar` 等），由多个分支分别修改，极易出现"部分状态已清除、部分状态仍残留"的不一致

用户反馈的"点击其他区域卡片飞到该位置"就是这种不一致的典型表现：`draggingPlanId` 还残留着非空值，但 `longPressTriggered` 已经是 false，后续的点击事件（新的 DOWN → UP）被识别为一次"拖拽"，导致卡片飞向点击位置。

---

#### 层次二：handleTaskBarDragEnd() 的状态清除未被可靠调用

**代码位置**：`handleTaskBarDragEnd()` 函数（~L301-343）

由于 `awaitEachGesture` 的循环条件是 `while (!event.changes.all { it.changedToUp() })`，只有当**所有指针**都抬起时才会退出循环并调用 `handleTaskBarDragEnd()`。但以下情况会导致 `handleTaskBarDragEnd()` **不被调用**：

1. **多指触摸干扰**：用户拖拽过程中第二根手指按下，然后第一根手指抬起 → `all { it.changedToUp() }` 为 false（还有第二根手指在按），循环继续但第二根手指的事件没有被正确追踪 → 手势状态永远卡在中间
2. **事件异常**：Compose 的手势系统因任何原因（窗口焦点变化、系统手势抢占等）终止事件流 → `awaitEachGesture` 协程被取消，`finally` 块中的状态清理不完整
3. **异常抛出**：循环内部的 `change.position`、`consume()` 等调用因空指针或类型转换异常抛出 → `handleTaskBarDragEnd()` 永远不会执行

这些场景都会导致：`draggingPlanId`、`isDraggingFromTaskBar`、`dragPosition` 等状态**永远不会被清空**，拖拽状态永久残留。

---

#### 层次三：draggingPlanId 残留导致的下游异常（用户可见现象）

当 `draggingPlanId != null` 但没有对应的手势更新时，会触发以下连锁反应：


```kotlin
// PinnedPlanItem 内部：isDragging=true 时 alpha 动画到 0f
```

同时保留 v1.2.7 中验证有效的修复：

## 问题四十六：任务栏排序后滚动到错误位置，被拖动卡片被挤出视口左侧（v1.2.9 第二轮修复尝试存在的问题）

### 问题现象

针对问题四十五（完全不滚动），v1.2.9 尝试引入滚动定位逻辑（`pendingScrollToPlanId` + `LaunchedEffect(pinnedPlans, pendingScrollToPlanId)`），但滚动问题仍然存在：

### 问题原因

**代码位置**：
- [HomeScreen.kt](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan/app/src/main/java/com/photo/plan/ui/home/HomeScreen.kt#L268-L284) 第 268-284 行 `LaunchedEffect(pinnedPlans, pendingScrollToPlanId)` 块

**问题分类**：第 3 类（Bug 原因与触发方式）+ 第 5 类（业务逻辑问题）

---

#### 第一层：scrollOffset 参数符号完全用反了（直接原因）

Jetpack Compose 的 `LazyListState.animateScrollToItem(index: Int, scrollOffset: Int)` API 的参数语义：

- `index`：目标 item 在 LazyList 中的索引
- `scrollOffset`：目标 item 的**起始边缘**相对于视口**起始边缘**的额外偏移量，单位是像素
  - `scrollOffset > 0`（正数）：item 的左边缘在视口左边缘的**右侧**（向内偏移，item 前面留出空白 padding）
  - `scrollOffset < 0`（负数）：item 的左边缘在视口左边缘的**左侧**（向外偏移，item 被推到视口外面去了）
  - `scrollOffset = 0`：item 的左边缘与视口左边缘精确对齐

当前错误实现：

```kotlin
// HomeScreen.kt 第 275-280 行
val cardPaddingPx = with(density) { 12.dp.toPx() }
try {
    taskBarListState.animateScrollToItem(
        index = targetIndex.coerceAtLeast(0),
        scrollOffset = -cardPaddingPx.toInt()  // ← 致命错误：传了负数！
    )
} catch (_: Exception) {}
```

`scrollOffset = -12dp` 表示让目标卡片的左边缘对齐到视口左边缘再往左 12dp 的位置。这会让目标卡片的左侧至少被挤出 12dp。

但这只能解释"卡片左边被挡住一点点"，无法解释"整个卡片都被挤出视口"的现象。整个卡片消失的根本原因在第二层。

---

#### 第二层：LaunchedEffect 触发时序与数据库异步更新不同步（根本原因）

`LaunchedEffect` 的 key 是 `pinnedPlans, pendingScrollToPlanId`。问题的触发涉及到 `pendingScrollToPlanId` 变化先于 `pinnedPlans` 更新的竞态条件。

以用户场景为例（拖动 index=2 的 -4 到 index=0），完整时序如下：

| 时间点 | 动作 | pinnedPlans 列表内容 | pendingScrollToPlanId 值 | LaunchedEffect 行为 |
|--------|------|----------------------|--------------------------|--------------------|
| **T0** | 用户松手，`handleTaskBarDragEnd()` 执行。第 334 行设置 `pendingScrollToPlanId = finalDraggingPlanId`，然后异步调用 `viewModel.reorderPinnedPlans()` 更新数据库 | `[-3, -2, -4, 202...]`（**旧数据**，-4 在 index=2） | 被设置为 `-4 的 id` | Compose 检测到 key 变化，但尚未调度 |
| **T1** | Compose 帧调度，`LaunchedEffect` **第一次触发** | 还是旧数据：`[-3, -2, -4, 202...]`（数据库 UPDATE 还没完成，Room Flow 还没发射新列表） | `-4 的 id`（非空） | 第 271 行：`targetIndex = pinnedPlans.indexOfFirst { it.id == targetId }` = **2**（**旧位置！**，-4 在旧列表的第 2 位） |
| **T2** | 执行第 273 行 | 旧数据 | **立即被置为 null** | |
| **T3** | 进入 `delay(80)`，协程挂起 80ms | 此时数据库可能正在执行 UPDATE，Room Flow 正在准备发射新列表 | null | |
| **T4** | `delay(80)` 结束，恢复执行 | **新数据已生效**：`[-4, -3, -2, 202...]`（-4 已经在 index=0） | null | **但 targetIndex 变量在 T1 阶段已经锁死为 2，不会重新计算！** 执行 `animateScrollToItem(index=2, scrollOffset=-12dp)` → 滚动到了新列表的 index=2（即 `-2` 卡片的位置），并且再往左偏移 12dp |
| **T5** | `pinnedPlans` 的值变化了，Compose 再次触发 `LaunchedEffect`（第二次） | `[-4, -3, -2, 202...]`（新数据） | **null**（T2 阶段已置空） | 第 270 行 `if (targetId != null)` 不成立，**什么都不做，直接退出** |

**最终效果**：
1. 实际滚动目标是新列表的 `index=2`（即 `-2` 卡片），而不是用户实际拖动的 `-4`
2. 再加上 `scrollOffset=-12dp` 把内容整体向左再推 12dp
3. 视口中能看到的最左边卡片是新列表的 `index=1`（即 `-3`）
4. 用户实际拖动的 `-4` 卡片在新列表的 `index=0`，完全被挤出了视窗左侧

这与用户截图反馈的现象**完全吻合**：松手后窗口只有 `-3` 停在视窗的最左边，`-4` 被挤出视窗左边，需要向左滑动才能看到。

### 问题设计缺陷总结

| 序号 | 缺陷点 | 代码位置 | 影响 |
|------|--------|----------|------|
| 1 | `scrollOffset` 参数符号用反，传了负值导致卡片被推到视口外 | 第 279 行 | 即使 index 正确，目标卡片左侧也会被遮挡 12dp |
| 2 | `pendingScrollToPlanId` 在第一次执行时立即被置为 null（第 273 行），导致 pinnedPlans 更新后的第二次 LaunchedEffect 触发什么都不做 | 第 273 行 | 根本没有机会用新数据重新计算正确的 targetIndex |
| 3 | `targetIndex` 在 `delay(80)` 之前就计算并锁死了，没有在 delay 之后（数据可能已更新）重新读取 pinnedPlans | 第 271 行 vs 第 277 行 | 滚动到了旧列表的 index，而不是新列表中目标 plan 的实际 index |
| 4 | LaunchedEffect 的两个 key（pinnedPlans 和 pendingScrollToPlanId）变化时机不同步，pendingScrollToPlanId 总是先变 | 第 268 行 | 第一次触发时数据永远是旧的，第二次触发时 pendingScrollToPlanId 已经被清空 |

### 正确的修复方向（待实施，当前版本暂不修改代码）

需要从以下几个维度同时修复：

**1. scrollOffset 取正值，留出合理的左侧 padding**

```kotlin
val cardPaddingPx = with(density) { 16.dp.toPx() }  // 16dp 更合理
// scrollOffset 应为正数：让卡片左边缘在视口左边缘右侧 16dp 处
scrollOffset = cardPaddingPx.toInt()
```

**2. targetIndex 必须在 delay 之后、滚动之前重新计算，确保使用最新的 pinnedPlans**

```kotlin
delay(80)
// 重新从最新的 pinnedPlans 中查找，不要用 delay 之前计算的旧值
val latestTargetIndex = pinnedPlans.indexOfFirst { it.id == targetId }
if (latestTargetIndex >= 0) {
    taskBarListState.animateScrollToItem(
        index = latestTargetIndex,
        scrollOffset = cardPaddingPx.toInt()
    )
}
```

**3. pendingScrollToPlanId 不要过早清空，等滚动完成后再置 null**

```kotlin
// 不要在 delay 之前就置 null
delay(80)
val latestTargetIndex = pinnedPlans.indexOfFirst { it.id == targetId }
if (latestTargetIndex >= 0) {
    taskBarListState.animateScrollToItem(...)
}
pendingScrollToPlanId = null  // 滚动完成后再清空
```

**4. 或者，改用更可靠的触发方式：用一个 Long 值的 version 计数器而不是直接用 planId 作为 key**

```kotlin
var scrollRequestVersion by remember { mutableStateOf(0L) }
var pendingScrollTargetId by remember { mutableStateOf<Long?>(null) }

// 触发滚动时：
pendingScrollTargetId = finalDraggingPlanId
scrollRequestVersion++  // 让 version 递增，确保每次都能触发 LaunchedEffect

// LaunchedEffect 用 version 作为唯一 key，每次递增都重新执行：
LaunchedEffect(scrollRequestVersion) {
    val targetId = pendingScrollTargetId
    if (targetId != null) {
        delay(100)  // 等待数据稳定
        val idx = pinnedPlans.indexOfFirst { it.id == targetId }
        if (idx >= 0) {
            val padding = with(density) { 16.dp.toPx() }
            taskBarListState.animateScrollToItem(index = idx, scrollOffset = padding.toInt())
        }
        pendingScrollTargetId = null
    }
}
```

这样可以彻底规避"两个 key 变化不同步"的竞态问题。

## Git 推送注意事项

GitHub Actions 工作流文件 [build-apk.yml](file:///remote-home/share/lijl/task_all/14_Photogragh-plan/.github/workflows/build-apk.yml) 的触发条件：

```yaml
on:
  push:
    tags:
      - 'v*'
```

**只有推送 `v*` 格式的 tag 才会触发 CI 构建和生成 Release APK**。每次推送时必须包含 tag 推送，完整指令如下：

```bash
# 假设当前版本号为 v1.2.9
cd /remote-home/share/lijl/task_all/14_Photogragh-plan/PhotoPlan

# 1. 如果本地和远程已有同名旧 tag，先删除（注意顺序：先删本地再删远程）
git tag -d v1.2.9
git push origin :refs/tags/v1.2.9

# 2. 提交代码
git add -A
git commit -m "fix: 修复任务栏拖拽排序相关问题"

# 3. 重新打 tag
git tag v1.2.9

# 4. 推送代码 + tag
git push origin master
git push origin v1.2.9
```

## 主流设计参考（任务栏拖拽排序）

市面上主流 App（iOS 主屏幕、Android 桌面、Trello、Notion、微信收藏夹）拖拽排序后的滚动行为：

1. **被拖动的卡片完整可见**：松手后，用户刚刚拖动的那张卡片必须 100% 完整地显示在视口中，不能有任何部分被遮挡
2. **合理的视觉边距**：卡片与视口边缘之间留有约 8~16dp 的安全边距，不会贴边
3. **平滑滚动动画**：使用 150~300ms 的平滑动画滚动到目标位置，而不是瞬间跳转
4. **滚动目标是用户实际操作的卡片**：绝不会滚动到其他卡片的位置
5. **滚动后用户无需额外手势**：用户不需要再手动滑动来查看结果
