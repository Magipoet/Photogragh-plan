# 摄影策划（PhotoPlan）项目构建问题记录

## 问题一：compileSdk 版本不匹配

### 问题现象

GitHub Actions 构建 APK 时，`checkDebugAarMetadata` 任务失败，报错退出码 1。关键错误信息：

```
Dependency 'androidx.core:core:1.15.0' requires libraries and applications that
depend on it to compile against version 35 or later of the Android APIs.
:app is currently compiled against android-34.

Dependency 'androidx.core:core-ktx:1.15.0' requires libraries and applications that
depend on it to compile against version 35 or later of the Android APIs.
:app is currently compiled against android-34.
```

### 问题原因

项目依赖的 `androidx.core:core:1.15.0` 和 `androidx.core:core-ktx:1.15.0` 要求 `compileSdk` 至少为 35，但 `app/build.gradle.kts` 中配置的 `compileSdk = 34`，版本不兼容导致构建失败。

### 解决方案

将 `PhotoPlan/app/build.gradle.kts` 中的 `compileSdk` 从 `34` 修改为 `35`。

```kotlin
// 修改前
compileSdk = 34

// 修改后
compileSdk = 35
```

### 解决效果

`checkDebugAarMetadata` 任务通过，构建流程进入下一阶段。

## 问题二：应用图标资源缺失

### 问题现象

修复 compileSdk 问题后，构建在 `processDebugResources` 阶段失败，AAPT 报错：

```
ERROR: AndroidManifest.xml:8:5-25:19: AAPT: error: resource mipmap/ic_launcher
(aka com.photo.plan:mipmap/ic_launcher) not found.
```

### 问题原因

`AndroidManifest.xml` 中声明了 `android:icon="@mipmap/ic_launcher"`，但项目的 `res/` 目录下完全没有 `mipmap` 目录和 `ic_launcher` 资源文件。AAPT 在链接资源时找不到该引用，直接报错。

### 解决方案

创建 Android Adaptive Icon 所需的三个 XML 资源文件：

1. `res/mipmap-anydpi-v26/ic_launcher.xml` — 自适应图标定义，引用前景和背景两层
2. `res/drawable/ic_launcher_background.xml` — 深绿色背景层
3. `res/drawable/ic_launcher_foreground.xml` — 白色相机图标前景层

由于项目 `minSdk = 26`，所有设备均支持 Adaptive Icon，因此只需 `mipmap-anydpi-v26` 即可。

### 解决效果

AAPT 资源链接通过，`processDebugResources` 任务成功，构建流程进入 Kotlin 编译阶段。

## 问题三：Kotlin 可空类型不匹配

### 问题现象

修复资源问题后，Kotlin 编译阶段失败，`compileDebugKotlin` 任务报错：

```
e: CreatePlanViewModel.kt:84:63 Argument type mismatch: actual type is 'kotlin.Long?',
   but 'kotlin.Long' was expected.
e: CreatePlanViewModel.kt:93:64 Argument type mismatch: actual type is 'kotlin.Long?',
   but 'kotlin.Long' was expected.
e: CreatePlanViewModel.kt:97:34 Argument type mismatch: actual type is 'kotlin.Long?',
   but 'kotlin.Long' was expected.
e: CreatePlanViewModel.kt:103:25 Argument type mismatch: actual type is 'kotlin.Long?',
   but 'kotlin.Long' was expected.
```

### 问题原因

原始代码在 `CreatePlanViewModel.kt` 的 `savePlan` 方法中：

```kotlin
val planId = if (_state.value.isEditMode && _state.value.planId != null) {
    val pid = _state.value.planId  // 此时 pid 仍为 Long?，智能转换失效
    val existing = planRepository.getPlanById(pid)!!
    ...
    pid  // Long? 类型
} else {
    planRepository.insertPlan(PlanEntity(name = name))
}
```

`_state` 是 `MutableStateFlow`，其 `value` 属性是可变的。虽然前面的条件判断了 `_state.value.planId != null`，但第 83 行重新读取 `_state.value.planId` 时，Kotlin 编译器无法保证两次读取之间值没有变化（另一个线程可能修改了 StateFlow 的值），因此**智能转换失效**，`pid` 仍然是 `Long?` 可空类型。后续 `pid` 被传入需要 `Long` 非空参数的函数，导致编译错误。

### 解决方案

重构条件判断逻辑，先将 `_state.value.planId` 读取到局部变量 `pid` 中，再用 `if (pid != null)` 进行判断。局部变量是不可变的，Kotlin 编译器可以正常进行智能转换，`pid` 在 `if` 块内自动变为 `Long` 非空类型：

```kotlin
val planId = if (_state.value.isEditMode) {
    val pid = _state.value.planId
    if (pid != null) {
        val existing = planRepository.getPlanById(pid)
        planRepository.updatePlan(
            existing.copy(name = name, updatedAt = System.currentTimeMillis())
        )
        pid  // 此处 pid 已被智能转换为 Long
    } else {
        planRepository.insertPlan(PlanEntity(name = name))
    }
} else {
    planRepository.insertPlan(PlanEntity(name = name))
}
```

### 解决效果

4 处类型不匹配错误全部消除，Kotlin 编译通过，项目构建成功，产出 `app-debug.apk` 和 `app-release-unsigned.apk`。

## 问题四：CI 产出两个 APK 文件

### 问题现象

构建成功后，GitHub Actions 产出两个 Artifact：`app-debug` 和 `app-release`。

### 原因说明

这是正常现象，不是错误。Android 项目默认有 `debug` 和 `release` 两种构建类型，CI 工作流 `build-apk.yml` 中配置了两次构建和两次上传：

- `./gradlew assembleDebug` → `app-debug.apk`（已签名，可直接安装调试）
- `./gradlew assembleRelease` → `app-release-unsigned.apk`（未签名，不可直接安装）

当前 `release` 构建未启用代码混淆（`isMinifyEnabled = false`）且未配置签名，与 debug 包内容差异很小，实际使用中 debug 包即可。release 构建步骤保留的价值在于提前发现 release 构建特有的问题。

**后续已优化**：删除了 release 构建，仅保留 debug 版本，简化 CI 流程。

## 问题五：安装时出现"解析软件包时出现问题 (33)"

### 问题现象

在手机上安装 APK 时弹出错误提示：
```
解析软件包时出现问题。(33)
详细信息: packageInfo is null
```

### 问题原因

两个独立问题共同导致：

1. **`AndroidManifest.xml` 缺失 `package` 属性**（主要原因）
   - 虽然 AGP 7.0+ 使用 `namespace` 定义包名，但 Android 系统解析 APK 时**必须**从 `AndroidManifest.xml` 的 `package` 属性读取包名以生成 `PackageInfo`
   - 缺少该属性导致系统解析失败，返回 `packageInfo is null`

2. **Release 构建 APK 未签名**
   - `app-release-unsigned.apk` 没有数字签名，Android 系统拒绝安装
   - 错误代码 33 对应 `INSTALL_PARSE_FAILED_BAD_PACKAGE_NAME`

### 解决方案

1. 在 `AndroidManifest.xml` 的 `<manifest>` 标签添加 `package="com.photo.plan"` 属性
2. 在 `build.gradle.kts` 中配置 release 构建使用 debug 签名：`signingConfig = signingConfigs.getByName("debug")`

### 解决效果

APK 可正常解析和安装。

## 问题六：选择参考图像后"保存策划"按钮灰色不可用

### 问题现象

从相册选择参考图像后，"保存策划"按钮一直是灰色的，无法点击。退出后图片也不会被保存，导致整个核心功能完全不可用。

### 问题原因

保存按钮的启用条件存在严重的逻辑错误。原代码如下：

```kotlin
Button(
    ...
    enabled = state.name.isNotBlank() && !state.isSaving,
)
```

按钮的启用条件**只检查"策划名称是否非空"**，完全没有检查"是否已选择图片"。这造成了致命的逻辑倒置：

- ❌ **选了图片但没填名称 → 按钮灰色**（用户的核心操作被忽视）
- ✅ **填了名称但没选图片 → 按钮可用**（可以保存空策划，毫无意义）

对于摄影策划软件，图片才是最核心的内容，名称只是辅助描述。

### 解决方案

1. 修改按钮启用条件为"至少有一张图片时可用"：

```kotlin
val hasImages = state.selectedUris.isNotEmpty() || state.existingSamples.isNotEmpty()
Button(
    ...
    enabled = hasImages && !state.isSaving,
)
```

2. 未填策划名称时点击保存，弹出提示对话框而非静默禁用：

```kotlin
if (name.isEmpty()) {
    _state.value = _state.value.copy(showNamePrompt = true)
    return
}
```

### 解决效果

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 选了图片，没填名称 | 按钮灰色 ❌ | 按钮可用，点击弹出提示 ✅ |
| 选了图片，填了名称 | 按钮可用 ✅ | 按钮可用 ✅ |
| 没选图片，填了名称 | 按钮可用（可保存空策划）❌ | 按钮灰色 ✅ |
| 没选图片，没填名称 | 按钮灰色 ✅ | 按钮灰色 ✅ |

## 问题七：图像查看器点击所有图片都显示第一张内容

### 问题现象

在"待拍摄"或"已完成"区域中，点击任何一张图片打开查看器，显示的都是第一张图片的内容，尽管封面缩略图是不同的。

### 问题原因

`ImageViewerScreen.kt` 中 `rememberPagerState` 的 `initialPage` 参数在首次组合时计算，但此时 `samples` 列表尚未加载完成（为空列表），导致 `initialPage` 始终被设为 0：

```kotlin
// 问题代码
val pagerState = rememberPagerState(
    initialPage = initialIndex.coerceAtMost((samples.size - 1).coerceAtLeast(0)),
    pageCount = { samples.size }
)
```

- 首次组合时：`samples` 为空 → `initialPage` = 0
- 数据加载完成后：`rememberPagerState` 不会重新使用 `initialPage`（它只在创建时使用一次）
- 结果：无论传入的 `initialIndex` 是多少，页面始终停留在第 0 页

### 解决方案

将 `initialPage` 设为 0，并添加 `LaunchedEffect` 监听数据加载完成，加载后跳转到正确的页面：

```kotlin
val pagerState = rememberPagerState(
    initialPage = 0,
    pageCount = { samples.size }
)

LaunchedEffect(samples.size) {
    if (samples.isNotEmpty()) {
        val targetPage = initialIndex.coerceIn(0, samples.size - 1)
        pagerState.scrollToPage(targetPage)
    }
}
```

### 解决效果

点击任何一张图片，查看器都会正确显示对应图片，并在顶部显示正确的页码（如 "3 / 10"）。

## 问题八：图像查看器不能左右滑动查看相邻图片

### 问题现象

在"待拍摄"和"已完成"区域点开图像后，无法左右滑动查看其他图片，只能退出后重新点开另一张，严重影响使用体验。

### 问题原因

代码中已使用了 `HorizontalPager`（理论上支持左右滑动），但 `ZoomableImage` 中的 `detectTransformGestures` 手势检测器存在阈值问题：

```kotlin
// 问题代码
if (scale > 1f) {
    offset = Offset(x = offset.x + pan.x, y = offset.y + pan.y)
} else {
    offset = Offset.Zero
}
```

- 当 `scale == 1f`（未缩放）时，轻微的触摸抖动可能使 `scale` 略微超过 1.0（如 1.0001）
- 此时 `detectTransformGestures` 开始消费平移事件用于图片拖动
- `HorizontalPager` 接收不到滑动事件，导致无法切换图片
- 另外，缩放时 offset 没有边界限制，可能导致图片被拖出可视区域

### 解决方案

1. 将缩放阈值从 `> 1f` 调整为 `> 1.01f`，留出安全边距防止误触
2. 添加 offset 边界限制，避免图片平移超出屏幕范围

```kotlin
detectTransformGestures { _, pan, zoom, _ ->
    val newScale = (scale * zoom).coerceIn(1f, 5f)
    scale = newScale
    if (newScale > 1.01f) {
        val maxX = (newScale - 1) * size.width / 2
        val maxY = (newScale - 1) * size.height / 2
        offset = Offset(
            x = (offset.x + pan.x).coerceIn(-maxX, maxX),
            y = (offset.y + pan.y).coerceIn(-maxY, maxY)
        )
    } else {
        offset = Offset.Zero
    }
}
```

### 交互逻辑

| 手势 | 缩放状态 | 行为 |
|------|----------|------|
| 单指左右滑动 | 未缩放（1x） | 切换上一张/下一张图片 |
| 双指缩放 | — | 放大/缩小图片 |
| 单指拖动 | 已缩放（>1x） | 平移图片 |
| 缩小至 1x | — | 恢复为滑动切换模式 |

## 问题九：CI 优化 — 删除冗余构建，改用 GitHub Release 发布

### 问题现象

1. CI 同时构建 `debug` 和 `release` 两个版本，但 release 版本未启用混淆且使用 debug 签名，与 debug 版本内容几乎相同，浪费 CI 资源
2. 构建产物存储在 GitHub Actions Artifacts 中，存储容量小（500MB 账号共享），易超限导致构建失败
3. `git push origin main` 会触发构建，导致频繁的无效构建

### 解决方案

1. **删除 release 构建**：仅保留 `assembleDebug`，删除 `assembleRelease` 及对应的上传步骤
2. **取消 Artifact 存储**：删除所有 `upload-artifact` 步骤
3. **改为 GitHub Release 发布**：使用 `softprops/action-gh-release@v2` 将 APK 发布到 GitHub Releases 页面
4. **仅 tag 触发构建**：CI 的 `on.push` 触发条件从 `branches: [main]` 改为 `tags: ['v*']`，`git push origin main` 只提交代码不构建

```yaml
on:
  push:
    tags:
      - 'v*'
  workflow_dispatch:
```

### 效果

| 项目 | 修改前 | 修改后 |
|------|--------|--------|
| 存储位置 | Actions Artifacts | GitHub Releases |
| 存储限制 | 500MB（账号共享，易超限） | 无限制 |
| 保留时间 | 90 天后自动删除 | 永久保留 |
| 构建触发 | push 到 main 即触发 | 仅推送 v* tag 触发 |
| 构建产物 | debug + release（冗余） | 仅 debug（精简） |
| APK 文件名 | app-debug.apk | photoplan-v1.0.0.apk |

## 问题十：GitHub Release 创建失败 (403 权限不足)

### 问题现象

推送 tag v1.0.0 触发 CI 构建后，`softprops/action-gh-release@v2` 步骤失败：

```
⚠️ GitHub release failed with status: 403
{"message":"Resource not accessible by integration","status":"403"}
```

### 问题原因

GitHub Actions 工作流中未显式声明 `permissions`，默认的 `GITHUB_TOKEN` 权限不足以创建 Release 和生成 Release Notes。`softprops/action-gh-release@v2` 需要 `contents: write` 权限才能完成以下操作：

- 创建 GitHub Release
- 上传 Release 附件（APK 文件）
- 生成 Release Notes

### 解决方案

在 `.github/workflows/build-apk.yml` 中添加 `permissions` 声明：

```yaml
permissions:
  contents: write
```

### 解决效果

`GITHUB_TOKEN` 获得 `contents: write` 权限，CI 可正常创建 Release 并上传 APK。

## 问题十一：参考样图选择区域只能单行显示，图片多时不便查看

### 问题现象

在新建/编辑策划页面，选好的参考样图排列成一行（`LazyRow`），当图片较多时需要横向滑动查看，操作不便。

### 问题原因

`CreatePlanScreen.kt` 中"已添加"和"待添加"图片区域使用了 `LazyRow`（横向单行滚动列表），所有图片只能排成一行，无法充分利用屏幕空间。

### 解决方案

1. 将 `LazyRow` 替换为 `FlowRow`（流式网格布局），图片自动换行排列
2. 内容区域添加 `verticalScroll(rememberScrollState())` 支持纵向滚动
3. 保存按钮固定在底部，内容区域使用 `weight(1f)` 占据剩余空间

```kotlin
FlowRow(
    horizontalArrangement = Arrangement.spacedBy(8.dp),
    verticalArrangement = Arrangement.spacedBy(8.dp)
) {
    state.existingSamples.forEach { sample ->
        ExistingImageItem(sample = sample, onRemove = { ... })
    }
}
```

### 解决效果

参考样图以多行网格方式排列，无需横向滑动即可一览所有图片。

## 问题十二：图像查看器无法左右滑动切换图片（手势冲突）

### 问题现象

在"待拍摄"区域点击图片查看大图后，无法左右滑动切换到其他参考样图的大图，只能退出后重新点开另一张。此问题在多个版本中反复出现。

### 问题原因

`ImageViewerScreen.kt` 中 `ZoomableImage` 的缩放手势检测器会消费所有触摸事件（包括单指水平滑动），导致 `HorizontalPager` 无法接收到滑动手势，左右翻页功能被完全拦截。

之前的 `detectTransformGestures` 方案仅通过调整缩放阈值（`> 1.01f`）无法根本解决问题，因为该检测器在任何情况下都会消费单指触摸事件。

### 解决方案

用自定义手势处理器替换 `detectTransformGestures`，核心逻辑：

- **双指触摸**：消费事件，处理缩放
- **单指触摸 + 已缩放**（scale > 1.01f）：消费事件，处理图片平移
- **单指触摸 + 未缩放**（scale = 1f）：**不消费事件**，让 `HorizontalPager` 处理左右翻页

使用 `awaitPointerEventScope` + `PointerEventType` 事件分类实现精确的手势分发控制。

### 交互逻辑

| 手势 | 缩放状态 | 行为 |
|------|----------|------|
| 单指左右滑动 | 未缩放（1x） | 切换上一张/下一张图片 |
| 双指缩放 | — | 放大/缩小图片 |
| 单指拖动 | 已缩放（>1x） | 平移图片 |
| 缩小至 1x | — | 恢复为滑动切换模式 |

## 问题十三：自定义手势处理器编译错误

### 问题现象

修复问题十二后，GitHub Actions 构建失败，Kotlin 编译器报出 20 余个错误：

```
e: Unresolved reference 'it'
e: Function invocation 'size(...)' expected
e: Unresolved reference 'position'
e: Unresolved reference 'getDistance'
e: Cannot infer type for this parameter
```

### 问题原因

手写手势处理代码时使用了不正确的 Compose 低级手势 API：

1. `activeChanges.size` 被误解析为函数调用而非属性访问
2. `Offset` 的 `getDistance()` 方法不存在（应使用 `distance` 属性或自定义计算）
3. Lambda 表达式中 `it` 引用和类型推断失败
4. `PointerInputChange` 的属性和方法调用方式有误

### 解决方案

完全重写手势处理器，使用正确的 Compose API：

1. 使用 `awaitPointerEventScope` + `awaitPointerEvent()` 获取事件
2. 使用 `PointerEventType.Press/Move/Release` 分类处理事件
3. 用 `event.changes.filter { c -> c.pressed }` 获取活跃触摸点
4. 使用 `pressedChanges[0].position` / `.previousPosition` 获取位置
5. 自定义 `distanceBetween()` 和 `centerOf()` 辅助函数替代不存在的 API

### 解决效果

编译错误全部消除，构建成功通过。

## 问题十四：大图查看时图片位置偏下

### 问题现象

在图像查看器中查看大图时，图片整体偏下，顶部留有较多空白。

### 问题原因

原来使用 `Scaffold` 包裹内容，`TopAppBar` 占据顶部空间后将剩余空间作为内容区域传递给 `Box`，图片只在 TopAppBar 下方的区域居中显示，视觉上整体偏下。

### 解决方案

改为 `Box` 全屏布局：

1. 图片 `HorizontalPager` 填满整个屏幕
2. `TopAppBar` 作为半透明叠层浮在图片上方
3. 使用 `statusBarsPadding()` 适配状态栏

```kotlin
Box(modifier = Modifier.fillMaxSize()) {
    HorizontalPager(...) { ... }  // 全屏图片
```

## 问题十六：上传样图后主界面显示"暂无样图"

### 问题现象

创建策划并上传样图后，返回软件主界面，策划卡片的进度显示"暂无样图"，但进入详情页可以看到样图。

### 问题原因

`HomeViewModel.kt` 中 `planProgressMap` 的更新仅在 `plans` Flow 发射时触发。但 `plans` Flow 仅监听 `plans` 表的变化（Room 的 Flow 查询只观察查询涉及的表）。

保存策划的执行顺序：
1. 插入 plan 记录 → `plans` Flow 触发 → 此时 samples 数量为 0 → 进度为 0
2. 逐个保存 samples → `plans` Flow **不再触发**（因为 plans 表没有变化）→ 进度永远停留在 0

### 解决方案

在 `SampleDao` 中新增全局 samples 变化的 Flow，然后在 `HomeViewModel` 中用 `combine` 合并两个 Flow：

```kotlin
// SampleDao.kt
@Query("SELECT COUNT(*) FROM samples")
fun getTotalSampleCount(): Flow<Int>

// HomeViewModel.kt
val plans: StateFlow<List<PlanEntity>> = combine(
    planRepository.getAllPlans(),
    sampleDao.getTotalSampleCount()
) { plans, _ -> plans }
    .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), emptyList())
```

当 samples 表发生任何变化时，`getTotalSampleCount()` 会重新发射，触发 `combine` 重新计算，进而重新查询每个 plan 的进度。

### 解决效果

上传样图后返回主界面，策划卡片正确显示进度（如 "3/5 已完成"），不再显示"暂无样图"。

## 问题十八：主界面进度显示不实时更新

### 问题现象

在策划详情页将 4 张图像标记为"已完成"后，返回主界面，策划卡片依然显示 "0/6" 而非 "4/6"，进度信息严重滞后。

### 问题原因

`HomeViewModel.kt` 中 `planProgressMap` 使用的是一次性 `suspend` 查询而非响应式 Flow：

```kotlin
// 问题代码
viewModelScope.launch {
    plans.collect { planList ->
        val progressMap = mutableMapOf<Long, Pair<Int, Int>>()
        planList.forEach { plan ->
            val total = sampleDao.getSampleCount(plan.id)       // 一次性查询
            val completed = sampleDao.getCompletedCount(plan.id) // 一次性查询
            progressMap[plan.id] = Pair(total, completed)
        }
        _planProgressMap.value = progressMap
    }
}
```

虽然 `plans` Flow 会触发重新查询，但仅当 `plans` 表发生变化时才会触发。`samples` 表的完成状态变更不会触发 `plans` Flow 重新发射，导致进度不会更新。

### 解决方案

使用 `flatMapLatest` + `combine` 组合响应式 Flow，利用 `SampleDao` 中已有的 Flow 版查询方法：

```kotlin
val planProgressMap: StateFlow<Map<Long, Pair<Int, Int>>> =
    plans
        .map { planList -> planList.map { it.id } }
        .flatMapLatest { planIds ->
            if (planIds.isEmpty()) {
                flowOf(emptyMap())
            } else {
                val countFlows = planIds.map { planId ->
                    combine(
                        sampleDao.getSampleCountByPlanId(planId),      // Flow<Int>
                        sampleDao.getCompletedCountByPlanId(planId)    // Flow<Int>
                    ) { total, completed ->
                        planId to (total to completed)
                    }
                }
                combine(countFlows) { pairs ->
                    pairs.associate { it }
                }
            }
        }
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), emptyMap())
```

### 解决效果

标记图像完成后返回主界面，策划卡片立即显示正确进度（如 "4/6"），进度信息与详情页完全同步。

## 问题二十一：多次选择图片时缺乏去重机制

### 问题现象

用户在新建/编辑策划页面分几次选择图片时，如果在不同次选择中选中了同一张图像，参考样图列表中会出现重复的图像，没有去重机制。

### 问题原因

`CreatePlanViewModel.kt` 中 `addUris` 方法直接将所有新选中的 URI 添加到列表中，没有任何去重检查：

```kotlin
// 问题代码
fun addUris(uris: List<Uri>) {
    val current = _state.value.selectedUris.toMutableList()
    current.addAll(uris)  // 直接添加，无去重
    _state.value = _state.value.copy(selectedUris = current)
}
```

### 解决方案

实现**三层去重**逻辑：

1. **新选择 URIs 自身去重**：`uris.distinctBy { it.toString() }`，防止同一次选择中重复
2. **与已选 URIs 去重**：比较 `Uri.toString()`，防止与之前选择的重复
3. **与已存在图像去重**：通过 `ContentResolver` 查询文件的**大小+名称**，与已保存到本地的文件比较

```kotlin
fun addUris(uris: List<Uri>) {
    viewModelScope.launch {
        val currentState = _state.value
        val current = currentState.selectedUris.toMutableList()
        val existing = currentState.existingSamples

        val distinctNewUris = uris.distinctBy { it.toString() }

        for (uri in distinctNewUris) {
            val isDuplicateInSelected = current.any { it.toString() == uri.toString() }
            if (isDuplicateInSelected) continue

            val isDuplicateInExisting = withContext(Dispatchers.IO) {
                isUriDuplicateOfExisting(uri, existing)
            }
            if (isDuplicateInExisting) continue

            current.add(uri)
        }

        _state.value = currentState.copy(selectedUris = current)
    }
}

private fun isUriDuplicateOfExisting(uri: Uri, existing: List<SampleEntity>): Boolean {
    return try {
        val context = getApplication<Application>()
        val cursor = context.contentResolver.query(uri, null, null, null, null)
        cursor?.use {
            if (it.moveToFirst()) {
                val sizeIndex = it.getColumnIndex(OpenableColumns.SIZE)
                val nameIndex = it.getColumnIndex(OpenableColumns.DISPLAY_NAME)
                if (sizeIndex != -1 && nameIndex != -1) {
                    val newSize = it.getLong(sizeIndex)
                    val newName = it.getString(nameIndex)
                    return existing.any { sample ->
                        val file = File(sample.localPath)
                        file.exists() && file.length() == newSize && file.name == newName
                    }
                }
            }
        }
        false
    } catch (e: Exception) {
        false
    }
}
```

### 去重逻辑

| 层级 | 去重方式 | 说明 |
|------|---------|------|
| 1 | `distinctBy { it.toString() }` | 同一次选择中的重复 URI |
| 2 | `Uri.toString()` 比较 | 与之前已选但未保存的 URI 比较 |
| 3 | 文件大小+文件名比较 | 与已保存到本地的图像比较（编辑模式下） |

### 解决效果

多次选择同一张图像时，重复图像会被自动过滤，参考样图列表中不会出现重复。

## 问题二十三：图像查看器缺少向下滑动退出功能

### 问题现象

在策划详情页点击图片进入大图查看模式后，只能通过左上角返回箭头退出，无法通过向下滑动手势退出大图模式，不符合主流图片查看器的交互习惯。

### 问题原因

`ImageViewerScreen.kt` 中 `ZoomableImage` 的手势处理器仅处理了双指缩放和单指平移（缩放状态下），没有实现单指向下滑动的退出逻辑。用户在未缩放状态下向下滑动时，手势被 `HorizontalPager` 忽略（垂直方向不是其负责的维度），也无法退出查看器。

### 解决方案

在 `ZoomableImage` 的 `pointerInput` 中增加垂直下滑手势识别和退出逻辑：

1. **方向判断**：通过比较 `\|dy\| > \|dx\|` 且 `dy > 20px` 识别为垂直向下拖动，避免与 HorizontalPager 的左右滑动冲突
2. **视觉反馈**：使用 `Animatable` 控制图片的 `translationY` 和背景透明度，图片跟随手指下移，背景逐渐变透明
3. **退出判断**：向下拖动超过阈值 200dp 松手后，启动下滑+淡出并行动画，动画结束后调用 `onDismiss()` 退出
4. **回弹复位**：未达阈值松手时，启动回弹动画平滑回到原位
5. **互不干扰**：仅在 `scale <= 1.01f`（未缩放状态）下启用该手势；放大后单指用于平移图片，双指用于缩放

```kotlin
val dragYAnimatable = remember { Animatable(0f) }
val bgAlphaAnimatable = remember { Animatable(1f) }

// Move 事件中：识别垂直拖动
if (isVerticalDrag && totalDy > 0f) {
    scope.launch {
        dragYAnimatable.snapTo(totalDy)
        bgAlphaAnimatable.snapTo((1f - totalDy / (size.height * 1.5f)).coerceIn(0.3f, 1f))
    }
    change.consume()
}

// Release 事件中：并行动画
if (currentDragY > dragThreshold) {
    scope.launch {
        val animJob1 = launch { dragYAnimatable.animateTo(size.height.toFloat(), tween(220)) }
        val animJob2 = launch { bgAlphaAnimatable.animateTo(0f, tween(220)) }
        joinAll(animJob1, animJob2)
        onDismiss()
    }
} else {
    scope.launch {
        val animJob1 = launch { dragYAnimatable.animateTo(0f, tween(200)) }
        val animJob2 = launch { bgAlphaAnimatable.animateTo(1f, tween(200)) }
        joinAll(animJob1, animJob2)
    }
}
```

### 解决效果

| 手势 | 行为 |
|------|------|
| 单指向下滑动 | 图片跟随手指下移，背景渐透明 |
| 下滑超过 200dp 后松手 | 图片继续下滑出屏幕 + 背景淡出 → 退出查看器 |
| 下滑不足 200dp 松手 | 图片平滑回弹复位，背景恢复不透明 |
| 双指缩放/单指平移（已放大） | 不触发退出逻辑 |

### 问题现象

在新建/编辑策划页面选择照片后，参考样图虽然每行排三张，但图片全部挤在屏幕左侧，右侧留有明显的空白区域，影响观感。

### 解决方案

1. `FlowRow` 添加 `modifier = Modifier.fillMaxWidth()` 和 `maxItemsInEachRow = 3`
2. 使用 `BoxWithConstraints` 获取父容器宽度，精确计算每张图片宽度为 `(maxWidth - 2 × 8dp间距) / 3`
3. 图片 Item 使用 `.width(itemWidth).aspectRatio(1f)` 替代 `.size(100.dp)`，确保三张图片均分整行

```kotlin
BoxWithConstraints(modifier = Modifier.fillMaxWidth()) {
    val itemSpacing = 8.dp
    val itemWidth = (maxWidth - itemSpacing * 2) / 3
    FlowRow(
        modifier = Modifier.fillMaxWidth(),
        horizontalArrangement = Arrangement.spacedBy(itemSpacing),
        verticalArrangement = Arrangement.spacedBy(itemSpacing),
        maxItemsInEachRow = 3
    ) {
        state.existingSamples.forEach { sample ->
            ExistingImageItem(
                sample = sample,
                itemWidth = itemWidth,
                onRemove = { viewModel.removeExistingSample(sample) }
            )
        }
    }
}
```

### 解决效果

参考样图每行恰好三张，均匀铺满屏幕宽度，无右侧留白。

## 问题二十五：向下滑动退出时松手后出现短暂空白界面

### 问题现象

在图像查看器中向下滑动退出时，手指离开屏幕后，能感到明显的停滞——图片下滑消失后出现一段空白界面（时间较短但很明显），然后才切回策划详情页。

### 问题原因

`ImageViewerScreen.kt` 中 `ZoomableImage` 的退出动画使用了**串行执行**的两个 `animateTo`：

```kotlin
// 问题代码
scope.launch {
    dragYAnimatable.animateTo(targetY, tween(250))   // 250ms：图片滑出屏幕
    bgAlphaAnimatable.animateTo(0f, tween(250))       // 又 250ms：背景淡出
    onDismiss()                                        // 总计 500ms 后才退出
}
```

Kotlin 协程中 `animateTo` 是挂起函数，第二个 `animateTo` 必须等第一个完成后才开始执行。时序问题如下：

1. **0~250ms**：图片从手指离开位置下滑到屏幕底部，同时背景从半透明渐变到更透明
2. **250~500ms**：图片已经完全不在屏幕上，但背景还在花 250ms 从约 0.3 透明度淡出到 0 —— 这段时间 ImageViewer 页面内容全透明，但页面还没退出，底下策划界面也还没出现
3. **500ms**：`onDismiss()` 终于执行，页面切换

关键问题在于第 2 阶段：图片已经滑走但页面还没退出的 250ms，用户看到的就是空白界面。

### 解决方案

将两个 `animateTo` 改为**并行执行**，用独立的子协程启动，`joinAll` 同时等待两者完成：

```kotlin
// 修复代码
scope.launch {
    val animJob1 = launch {
        dragYAnimatable.animateTo(size.height.toFloat(), tween(220))
    }
    val animJob2 = launch {
        bgAlphaAnimatable.animateTo(0f, tween(220))
    }
    joinAll(animJob1, animJob2)
    onDismiss()
}
```

两个动画同时进行，总耗时从 500ms 降为 220ms，且背景淡出和图片下滑同步完成，不存在图片已消失但背景还在淡出的空白期。同理，未达阈值的回弹动画也改为并行执行。

### 解决效果

| 阶段 | 修复前 | 修复后 |
|------|--------|--------|
| 下滑退出动画 | 串行 250ms + 250ms = 500ms | 并行 220ms |
| 松手后空白期 | 约 250ms（图片已走、背景未退） | 无（图片和背景同步消失） |
| 回弹复位动画 | 串行 200ms + 200ms | 并行 200ms |
